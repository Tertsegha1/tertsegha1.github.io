PostgreSQL Process/Memory/Storage Model

~60 mins

The Postmaster and Backend Processes

When PostgreSQL starts, a single postmaster process spawns one backend process per client connection. Each backend handles SQL parsing, planning, and execution for its connection.

Memory Architecture

ComponentPurposeConfig parameter
shared_buffersShared page cache (all backends)shared_buffers
work_memPer-sort/hash operation memorywork_mem
wal_buffersWAL write bufferwal_buffers
maintenance_work_memVACUUM, CREATE INDEXmaintenance_work_mem
What each memory area actually does
shared_buffers is a pool shared by every connection. When PostgreSQL reads data from disk it stores a copy here, so the next query gets it from RAM instead of reading disk again — think of it as a shared filing cabinet the whole office can reach into. It is the most important parameter to tune: the more data you keep in this cache, the fewer slow disk reads happen. Target 25% of total RAM (e.g. 4 GB on a 16 GB server).

work_mem is the memory budget for a single sort or hash operation — ORDER BY, JOIN, DISTINCT each get their own slice. The catch: with 14 participants connected and each running complex queries, work_mem is multiplied across all concurrent operations. Set it too high and you exhaust RAM. The default (4 MB) is conservative; raise it only when slow sorts are confirmed.

wal_buffers is a small staging area where WAL entries are assembled in memory before being flushed to the WAL log file on disk — like a notepad you scribble on before writing the final record. PostgreSQL tunes this automatically so it rarely needs manual adjustment.

maintenance_work_mem is a separate budget reserved for maintenance operations — VACUUM cleaning up dead tuples and CREATE INDEX building a new index from scratch. Giving maintenance its own allocation keeps it from competing with live queries for memory.

Storage Layout

Each table stored in 8KB pages. Each page contains tuples. Dead tuples from UPDATE/DELETE are left in place and reclaimed by VACUUM.

Pages, tuples, and dead tuples explained
8KB pages: PostgreSQL never reads individual rows from disk — it reads entire 8-kilobyte blocks called pages. Each page is a fixed-size chunk of a table's file on disk. When you query one row, PostgreSQL loads the whole page that contains it into shared_buffers. This is why shared_buffers size matters so much: if the page is already cached, the query is fast; if it is not, PostgreSQL must go to disk.

Tuples: a tuple is one row in a table. Each page holds as many tuples as fit within 8 KB. Every tuple carries two hidden system columns — xmin (the transaction that created it) and xmax (the transaction that deleted or replaced it). These are the numbers MVCC uses to decide whether a row is visible to a given query.

Dead tuples: when you UPDATE a row, PostgreSQL does not overwrite it — it writes a new version and marks the old version expired by setting xmax. The old version is now a dead tuple: invisible to new queries but still sitting on disk taking up space. PostgreSQL leaves it there deliberately because a transaction that started before the update may still need to see it. Once no active transaction can possibly need the old version, VACUUM marks that space as reusable. A table that is updated heavily without regular VACUUM grows larger on disk even if the row count stays the same — this is called table bloat, and Day 6 covers how to detect and fix it.

Write-Ahead Logging (WAL)

~40 mins

WAL is the mechanism that delivers the Durability guarantee in ACID. Before any data page is written to disk, the change is recorded in the WAL log. On crash, PostgreSQL replays WAL to reach a consistent state.

Key insight
Sequential WAL writes are much faster than random data-page writes. This is why WAL enables high throughput even on spinning disks.

Key WAL Settings

sql
SELECT name, setting, unit
FROM pg_settings
WHERE name IN ('wal_level','synchronous_commit','max_wal_size',
             'checkpoint_completion_target','wal_compression')
ORDER BY name;
What these settings control
wal_level: minimum is replica for streaming replication; logical adds logical decoding. synchronous_commit: on = fsync WAL before confirming commit (safe); off = async commit, up to ~600ms data loss on crash but ~3× write throughput — only use for non-critical bulk loads. max_wal_size: how large WAL can grow between checkpoints; increasing this reduces checkpoint frequency and I/O spikes at the cost of longer crash recovery. checkpoint_completion_target: spreads dirty-page writes over this fraction of the checkpoint interval — 0.9 means spread over 90% to avoid a sudden I/O spike. wal_compression: compresses full-page writes in WAL — free CPU↔I/O trade that almost always wins on modern hardware.

Multi-Version Concurrency Control (MVCC)

~40 mins

MVCC solves the readers-writers problem without locks. When a row is updated, PostgreSQL creates a new version with a higher transaction ID (xid). Old versions remain visible to older transactions; newer versions are visible only after commit.

MVCC conceptWhat it means
xminTransaction that inserted this row version
xmaxTransaction that deleted/updated (0 = live)
Snapshot isolationReaders see a consistent point-in-time view
Dead tupleRow version no longer visible to any transaction
VACUUMReclaims space from dead tuples
Why MVCC matters for NSR
In a social registry, caseworkers in Kano and Lagos may read and update the same household record simultaneously. Without MVCC, you would need table or row locks, meaning Reader A's SELECT would block Writer B's UPDATE or vice versa. MVCC eliminates that: readers always get a consistent snapshot of the data as it existed at the start of their transaction, while writers create new row versions in parallel. The practical result — no read/write deadlocks, predictable query latency, and audit-safe reads even during bulk data corrections. The cost is dead tuples that VACUUM must reclaim; that's the WAL → MVCC → VACUUM triad every participant should be able to explain by end of Day 1.

Normalisation & NSR Schema Design

~50 mins

The NSR (National Social Registry) requires a normalised schema to avoid redundancy, enforce integrity, and support audit. The target is Third Normal Form (3NF).

3NF Rule

Every non-key column must depend ONLY on the primary key — not on another non-key column (no transitive dependency).

Core NSR Tables

sql – NSR schema
CREATE TABLE nsr.states (
  state_code  CHAR(2)    PRIMARY KEY,
  state_name  TEXT       NOT NULL UNIQUE
);

CREATE TABLE nsr.lgas (
  lga_id      SERIAL     PRIMARY KEY,
  state_code  CHAR(2)    NOT NULL REFERENCES nsr.states(state_code),
  lga_name    TEXT       NOT NULL
);

CREATE TABLE nsr.households (
  household_id      BIGSERIAL  PRIMARY KEY,
  household_code    TEXT       NOT NULL UNIQUE,
  lga_id            INT        NOT NULL REFERENCES nsr.lgas(lga_id),
  head_name         TEXT       NOT NULL,
  pmt_score         NUMERIC(8,3),
  enumeration_round SMALLINT,
  is_active         BOOLEAN    NOT NULL DEFAULT TRUE
);

CREATE TABLE nsr.individuals (
  individual_id  BIGSERIAL  PRIMARY KEY,
  household_id   BIGINT     NOT NULL REFERENCES nsr.households(household_id)
                            ON DELETE CASCADE,
  nin            CHAR(11)   UNIQUE
                 CHECK (nin IS NULL OR nin ~ '^[A-Z]{2}[0-9]{7}[A-Z]{2}$'),
  full_name      TEXT       NOT NULL,
  dob            DATE       CHECK (dob > '1900-01-01' AND dob <= CURRENT_DATE),
  gender         CHAR(1)    CHECK (gender IN ('M','F','O'))
);
Design decisions worth calling out
BIGSERIAL vs SERIAL: the NSR can grow to tens of millions of individuals; BIGSERIAL (64-bit) avoids integer overflow at ~2.1 billion rows that SERIAL would hit. ON DELETE CASCADE on individuals: if a household is removed (e.g. duplicate detected), all members cascade — prevents orphaned rows without a manual DELETE. NIN regex CHECK: '^[A-Z]{2}[0-9]{7}[A-Z]{2}$' enforces NIMC format at the database level, not just the application layer — the registry is the authoritative source so data quality is enforced here, not in code that may be bypassed. dob range CHECK: rejects dates before 1900 and future dates — catches common data-entry errors in field enumeration. NULLABLE nin: many enumerated individuals, especially children, may not yet have a NIN; UNIQUE with NULL allowed means multiple rows can have NULL without conflicting (PostgreSQL treats NULLs as distinct in UNIQUE indexes).

Advanced — WAL Internals: LSN, Segments, and Crash Recovery

Every WAL write is stamped with a Log Sequence Number (LSN) — a monotonically increasing byte offset into the WAL stream. LSNs are the spine of crash recovery and replication: the primary sends WAL bytes from a known LSN; the standby applies them in order.

sql – inspecting WAL position live
-- Current WAL write position (LSN)
SELECT pg_current_wal_lsn()                     AS current_lsn,
       pg_walfile_name(pg_current_wal_lsn())     AS wal_file,
       pg_current_wal_insert_lsn()               AS insert_lsn;

-- Run a DML and watch LSN advance
SELECT pg_current_wal_lsn();        -- record before
UPDATE nsr.households SET is_active = TRUE WHERE household_id = 1;
SELECT pg_current_wal_lsn();        -- record after — delta = WAL bytes written

-- Distance between two LSNs in bytes
SELECT pg_wal_lsn_diff(
  pg_current_wal_lsn(),
  '0/01000000'    -- replace with your before-LSN
) AS bytes_since_start;

-- Last checkpoint info
SELECT checkpoint_lsn, prior_checkpoint_lsn,
       redo_lsn, wal_level, timeline_id
FROM pg_control_checkpoint();
Why LSNs matter in production NSR operations
WAL segments: WAL is written to 16 MB files (default) named with LSN offsets, e.g. 000000010000000000000001. These are the files pg_basebackup and streaming replication transfer. Crash recovery: on restart after a crash, PostgreSQL reads the pg_control file to find the last checkpoint LSN, then replays all WAL records from that point forward — this is why commits are durable even if data pages were not flushed. pg_wal_lsn_diff: measuring LSN distance before and after a large batch INSERT tells you exactly how many WAL bytes that operation generated — critical for capacity planning when replication lag is a concern. wal_level: must be at least replica for streaming replication, and logical for logical decoding / CDC pipelines.

Demo 1.5 – MVCC Two-Terminal Visibility

What you will observe
This demo requires two separate psql terminal windows open simultaneously. You will see that Terminal A (Lagos) holds a snapshot of the data taken at BEGIN — and that snapshot does not change even after Terminal B (Kano) commits an update. This is MVCC snapshot isolation. Follow each step in order; do not rush ahead.

Step 1 — Terminal A (Lagos caseworker): open a REPEATABLE READ transaction and read the row

sql – Terminal A (Lagos caseworker)
-- IMPORTANT: use REPEATABLE READ so the snapshot is fixed for the whole transaction
BEGIN TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT household_id, head_name, pmt_score
FROM  nsr.households
WHERE household_code = 'HH-LA-001';
Expected output — Terminal A
household_id | head_name | pmt_score
 1 | Fatima Yusuf | 28.500

Leave this transaction open. Do not type anything else in Terminal A yet.

Step 2 — Terminal B (Kano caseworker): commit an update to the same row

sql – Terminal B (Kano caseworker)
BEGIN;
UPDATE nsr.households
SET   pmt_score = 42.5
WHERE household_code = 'HH-LA-001';
COMMIT;
Expected output — Terminal B
UPDATE 1

Terminal B has committed. The value 42.5 is now in the database. Before reading on — predict what Terminal A will show if you run the SELECT again now.

Step 3 — Back to Terminal A: re-run the same SELECT

sql – Terminal A (Lagos caseworker)
-- Re-run the same SELECT while Terminal A's transaction is still open
SELECT household_id, head_name, pmt_score
FROM  nsr.households
WHERE household_code = 'HH-LA-001';
Expected output — Terminal A still shows 28.5
household_id | head_name | pmt_score
 1 | Fatima Yusuf | 28.500

Even though Terminal B committed 42.5, Terminal A's snapshot was taken at BEGIN — it sees the world as it was at that moment. This is exactly what MVCC guarantees.

Step 4 — Close Terminal A's transaction and re-read

sql – Terminal A
COMMIT;
-- Now open a fresh transaction and re-read
SELECT household_id, head_name, pmt_score
FROM  nsr.households
WHERE household_code = 'HH-LA-001';
Expected output — Terminal A now sees 42.5
household_id | head_name | pmt_score
 1 | Fatima Yusuf | 42.500

The new transaction opens a fresh snapshot that sees the committed world. The MVCC isolation was transaction-scoped, not connection-scoped.
Why this matters in NSR operations
A batch reporting job that opens a single transaction at midnight and runs for 45 minutes will see a perfectly consistent snapshot of every household — even if caseworkers are updating records throughout. The report is not distorted by half-completed changes. This is the operational value of snapshot isolation: reproducible, audit-safe reads under concurrent write load.

Advanced — Inspecting MVCC Internals with System Columns

Every PostgreSQL row carries hidden system columns. Querying them reveals exactly which transaction created and deleted each version — the raw mechanics of MVCC made visible.

sql – xmin, xmax and row visibility
-- See system columns on a live row
SELECT xmin, xmax, ctid, household_code, pmt_score
FROM  nsr.households
WHERE household_code = 'HH-LA-001';
-- xmin = transaction that INSERTed this row version
-- xmax = transaction that deleted/replaced it (0 means still live)
-- ctid  = (page, tuple) physical address — changes on every UPDATE

-- Run an UPDATE and observe the new version
BEGIN;
UPDATE nsr.households SET pmt_score = 35.0 WHERE household_code = 'HH-LA-001';

-- Inside the same transaction: what does xmin/xmax look like now?
SELECT xmin, xmax, ctid, pmt_score FROM nsr.households
WHERE household_code = 'HH-LA-001';
-- xmin = this transaction ID, ctid points to the NEW tuple location

ROLLBACK;

-- After rollback: xmin/xmax revert to the original row (old version becomes live again)
SELECT xmin, xmax, ctid, pmt_score FROM nsr.households
WHERE household_code = 'HH-LA-001';

-- Check the current transaction ID for reference
SELECT txid_current() AS my_txid;
What xmin and xmax reveal about a row's lifecycle
xmin: the transaction ID that created this row version. A row is visible to a snapshot only if xmin's transaction committed before the snapshot was taken. xmax: the transaction that marked this row version as superseded — either a DELETE or an UPDATE that wrote a new version. If xmax = 0, the row is live. If xmax is a committed transaction, the row is a dead tuple waiting for VACUUM. ctid: the physical (page, tuple) address. Because UPDATE writes a new version rather than modifying in place, ctid changes on every UPDATE — this is why PostgreSQL cannot use the physical address as a stable row pointer (unlike SQL Server's RID). Rollback behaviour: PostgreSQL does not undo WAL on rollback. The new row version is simply marked with xmax = the rolled-back transaction ID, which future readers will see as aborted and skip. No data is physically removed — VACUUM will clean it up.

Design an audit log table for NSR data changes
Intermediate ▼

The NSR compliance team requires a full audit trail of every change to nsr.households.pmt_score. For each change they need to know: which household changed, the old value, the new value, who made the change, and when.

Your task:

  1. Design and create a table nsr.pmt_score_audit to hold this information. Choose appropriate data types and add a primary key.
  2. Write a test INSERT into the audit table (simulate a score change for household_id = 1 from 28.5 to 42.5).
  3. Query the audit table to retrieve the full history for household_id = 1, ordered by change time descending.
Guidance — think about these before writing SQL
What data type stores a timestamp with time zone? How will you record who made the change — should it be a free-text name or the database username? Should old_score and new_score allow NULL (what if this is the first time a score is set)? Do you need a foreign key to households?
Write your DDL and test queries here
Reference Solution
sql – solution
-- Step 1: create the audit table
CREATE TABLE nsr.pmt_score_audit (
  audit_id      BIGSERIAL         PRIMARY KEY,
  household_id  BIGINT            NOT NULL REFERENCES nsr.households(household_id),
  old_score     NUMERIC(8,3),
  new_score     NUMERIC(8,3)      NOT NULL,
  changed_by    TEXT             NOT NULL DEFAULT current_user,
  changed_at    TIMESTAMPTZ      NOT NULL DEFAULT now()
);

-- Step 2: simulate a score change
INSERT INTO nsr.pmt_score_audit (household_id, old_score, new_score)
VALUES (1, 28.5, 42.5);

-- Step 3: view full history for household 1
SELECT
  a.audit_id,
  h.household_code,
  a.old_score,
  a.new_score,
  a.changed_by,
  a.changed_at
FROM  nsr.pmt_score_audit a
JOIN  nsr.households h ON a.household_id = h.household_id
WHERE a.household_id = 1
ORDER BY a.changed_at DESC;
Key design decisions
TIMESTAMPTZ not TIMESTAMP: always use TIMESTAMPTZ (timestamp with time zone) for audit columns — it stores the UTC offset so records remain unambiguous across time zone changes or server migrations.

old_score is nullable: the first time a score is ever set, there is no previous value. NULL correctly represents "did not exist before."

DEFAULT current_user: PostgreSQL's built-in function records the database login name automatically — no application code needed to populate it. This cannot be spoofed by a client that connects directly via psql.

Extension: in production, this audit insert would be triggered automatically by a PostgreSQL trigger on nsr.households — Day 3 covers triggers and row-level security.

Basics — What is DML and Why Does It Matter?

DML (Data Manipulation Language) is the set of SQL statements that change data: INSERT, UPDATE, DELETE, and TRUNCATE. Every payment record, household enrolment, and audit entry in the NSR is created and maintained through DML. Unlike DDL (which changes structure), DML always runs inside a transaction and can be rolled back.

sql – the four DML statements at a glance
-- INSERT: add a new row (ON CONFLICT DO NOTHING = safe to re-run if state already exists)
INSERT INTO nsr.states (state_code, state_name) VALUES ('OY', 'Oyo')
ON CONFLICT DO NOTHING;

-- UPDATE: change values in existing rows
UPDATE nsr.households SET is_active = FALSE WHERE household_code = 'HH-LA-001';

-- DELETE: remove specific rows (respects FKs, generates WAL, can be rolled back)
DELETE FROM nsr.households WHERE household_code = 'HH-XX-999';

-- TRUNCATE: remove ALL rows fast (no row-by-row WAL, resets sequences)
-- Create the staging table first if it doesn't exist yet
CREATE TABLE IF NOT EXISTS nsr.stg_households (
  household_code TEXT, state_name TEXT, head_name TEXT, pmt_score TEXT
);
TRUNCATE nsr.stg_households RESTART IDENTITY;

-- Always verify what you changed — COMMIT only when output looks right
BEGIN;
UPDATE nsr.households SET pmt_score = 28.5 WHERE household_id = 1;
SELECT household_id, pmt_score FROM nsr.households WHERE household_id = 1;
COMMIT; -- or ROLLBACK if the SELECT looks wrong
DELETE vs TRUNCATE — choose deliberately
DELETE removes rows one by one, fires triggers, respects FK constraints, and writes one WAL record per row — it is slow on large tables but safe and reversible. TRUNCATE removes all rows in a single operation, resets sequences if you use RESTART IDENTITY, and writes far less WAL — but it acquires an ACCESS EXCLUSIVE lock (blocks all other access) and does not fire row-level triggers. For a production NSR table with active users, TRUNCATE is dangerous; for a staging table between import runs it is the right choice. Always run TRUNCATE in a transaction you can roll back until you are certain.

Intermediate — INSERT / UPDATE / DELETE / TRUNCATE

sql – DML
-- Clean slate first so this block is safe to re-run
DELETE FROM nsr.households WHERE household_code = 'HH-WS-001';

-- INSERT with RETURNING
INSERT INTO nsr.households (household_code, lga_id, head_name, pmt_score)

RETURNING household_id, household_code;

WHERE household_code = 'HH-WS-001'
RETURNING household_id, household_code, pmt_score;

-- DELETE: confirm first (best practice)

DELETE FROM nsr.households WHERE household_code = 'HH-WS-001';

-- TRUNCATE resets sequence too
-- TRUNCATE nsr.households RESTART IDENTITY CASCADE;
Key points for each operation
RETURNING: unique to PostgreSQL — the INSERT or UPDATE returns columns from the affected row immediately, without a second SELECT. Invaluable in NSR workflows where you need the generated household_id (a BIGSERIAL) to insert related individuals in the same script. DELETE — always SELECT first: the pattern of running a SELECT with the same WHERE clause before deleting is not just a habit — it is the standard safeguard against an accidental mass delete. In the NSR, a wrong WHERE clause on the households table could cascade-delete thousands of individual records. TRUNCATE vs DELETE: TRUNCATE removes all rows instantly by deallocating storage pages — it cannot be rolled back outside a transaction, and RESTART IDENTITY resets sequences back to 1. It is appropriate for clearing staging tables between import batches, never for production data. CASCADE extends the truncate to child tables that reference this one via FK.

Basics — What a JOIN Does

A JOIN combines rows from two or more tables based on a related column. Without JOINs, you would have to write separate queries and piece the results together in code. Think of it as the database's way of answering: "give me this data from Table A together with related data from Table B, matched on a common key."

-- INNER JOIN: households where the LGA exists in our reference table
SELECT h.household_code, l.lga_name, h.pmt_score
FROM  nsr.households h
JOIN  nsr.lgas       l ON l.lga_id = h.lga_id
LIMIT 10;

SELECT l.lga_name, COUNT(h.household_id) AS household_count
FROM  nsr.lgas       l
LEFT JOIN nsr.households h ON h.lga_id = l.lga_id
GROUP BY l.lga_name
ORDER BY household_count DESC;
The most common JOIN mistake
Putting a filter on the right-side table in the WHERE clause of a LEFT JOIN silently converts it into an INNER JOIN — rows with no match are eliminated because WHERE NULL = value is always false. Always put right-side filters in the ON clause to preserve the outer-join behaviour. You will see a practical example of this in the Payment Status section below.

INNER JOIN vs LEFT JOIN — when to use each
INNER JOIN (the first query): returns only households that have a matching LGA and state. This is correct here because every household must belong to a valid LGA — if a row had no match it would indicate a data integrity problem, and silently hiding it in the results would be wrong. LEFT JOIN (the second query): returns every active household regardless of whether they have any payments. Households with zero payments appear with payment_count = 0 and total_paid = NULL — which is exactly the policy insight needed: who hasn't been paid yet? The AND in the LEFT JOIN ON clause: filtering on p.status = 'processed' inside the ON (not the WHERE) means "join only to processed payments." This preserves households with only pending/failed payments as non-matched rows, keeping them in the result with NULL values. Moving this condition to WHERE would silently drop those households from the output.

Step 2 — add payment aggregates with LEFT JOIN (preserves states with no payments)

SELECT
  s.state_name,
  COUNT(DISTINCT h.household_id)                         AS total_households,
  COUNT(DISTINCT CASE WHEN p.status = 'processed'
        THEN h.household_id END)                           AS households_paid,
  COALESCE(SUM(CASE WHEN p.status = 'processed'
        THEN p.amount_ngn END), 0)                        AS total_disbursed_ngn,
  COUNT(CASE WHEN p.status = 'failed'
        THEN 1 END)                                        AS failed_payments,
  ROUND(
    100.0 * COUNT(CASE WHEN p.status = 'failed' THEN 1 END)
           / NULLIF(COUNT(p.payment_id), 0), 1
  )                                                         AS failure_rate_pct
FROM  nsr.households h
JOIN  nsr.lgas   l ON h.lga_id     = l.lga_id
JOIN  nsr.states s ON l.state_code = s.state_code
LEFT JOIN nsr.payments p ON p.household_id = h.household_id
WHERE h.is_active = TRUE
GROUP BY s.state_name
ORDER BY total_disbursed_ngn DESC;
New patterns in this query — read before running
COUNT(DISTINCT CASE WHEN … THEN household_id END): counts unique household IDs only when the CASE condition is true — a standard pattern for "how many distinct X meet condition Y" without a subquery.

COALESCE(SUM(…), 0): SUM over an empty set returns NULL. COALESCE replaces NULL with 0 so the report shows a real number for states with no disbursements.

NULLIF(COUNT(p.payment_id), 0): dividing by zero causes an error. NULLIF returns NULL when the denominator is zero, making the division return NULL instead of crashing — then ROUND handles NULL safely by returning NULL.

Basics — Window Functions vs GROUP BY

A window function computes a value for each row using a "window" of related rows — without collapsing those rows into one. This is the key difference from GROUP BY: GROUP BY returns one row per group; a window function returns one row per input row, with the aggregate attached.

ApproachOutput rowsUse when you need
GROUP BY + COUNTOne per groupSummary totals only
COUNT() OVER ()One per input rowSummary alongside detail
sql – the simplest window: total count alongside each row
-- state_code lives in nsr.lgas, not nsr.households — always JOIN to get it

-- Without window: one summary row per state
SELECT l.state_code, COUNT(*) AS total
FROM  nsr.households h
JOIN  nsr.lgas l ON l.lga_id = h.lga_id
GROUP BY l.state_code
ORDER BY l.state_code;

-- With window: each household row PLUS its state's total
SELECT
  h.household_code,
  l.state_code,
  h.pmt_score,
  COUNT(*) OVER (PARTITION BY l.state_code) AS state_total,
  RANK()   OVER (PARTITION BY l.state_code ORDER BY h.pmt_score) AS poverty_rank
FROM  nsr.households h
JOIN  nsr.lgas l ON l.lga_id = h.lga_id
ORDER BY l.state_code, poverty_rank;
Anatomy of a window function call
function() OVER (PARTITION BY col ORDER BY col ROWS BETWEEN ...) — the OVER clause is what makes it a window function. PARTITION BY divides rows into groups (like GROUP BY, but doesn't collapse). ORDER BY inside OVER sets the ranking or frame order. ROWS BETWEEN defines the exact frame of rows to include in the calculation. If you omit PARTITION BY, the entire result set is one window.

Intermediate — Window Functions

Window functions compute values across a set of rows related to the current row — without collapsing them like GROUP BY does.

sql – window functions
-- Rank households by pmt_score within their state
SELECT
  h.household_code,
  h.head_name,
  s.state_name,
  h.pmt_score,
  RANK() OVER (
    PARTITION BY l.state_code
    ORDER BY h.pmt_score ASC
  ) AS poverty_rank,
  ROUND(
    (100.0 * PERCENT_RANK() OVER (
      PARTITION BY l.state_code ORDER BY h.pmt_score
    ))::NUMERIC, 1
  ) AS percentile
FROM nsr.households h
JOIN  nsr.lgas   l ON h.lga_id     = l.lga_id
JOIN  nsr.states s ON l.state_code = s.state_code
WHERE h.is_active = TRUE
ORDER BY s.state_name, poverty_rank;

-- Running total of payments per household
SELECT
  household_id,
  payment_date,
  amount_ngn,
  SUM(amount_ngn) OVER (
    PARTITION BY household_id
    ORDER BY payment_date
    ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
  ) AS cumulative_paid
FROM nsr.payments
WHERE status = 'processed'
ORDER BY household_id, payment_date;
How window functions differ from GROUP BY
GROUP BY collapses rows — one output row per group, detail lost. Window functions keep every row and attach a computed value alongside it. The OVER () clause defines the window: PARTITION BY divides rows into independent groups (here, per state) and ORDER BY controls ranking or accumulation order within each group. RANK() vs DENSE_RANK(): if two households tie at rank 3, RANK() produces 3, 3, 5 (skips 4); DENSE_RANK() produces 3, 3, 4 (no gap). For poverty targeting, RANK() is more conservative — tied households are treated equally and the next distinct score gets a lower priority number. PERCENT_RANK(): returns a value from 0 to 1 representing where the row falls relative to all rows in the partition — multiplied by 100 it gives the percentile. A household at the 10th percentile is among the most vulnerable 10% in its state. ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW: the explicit frame clause for the running total — includes all rows from the start of the partition up to and including the current row, ordered by payment_date, giving a cumulative disbursement total per household.

LAG and LEAD — payment trend analysis

LAG() looks back at the previous row in the window; LEAD() looks forward. Both are essential for detecting changes between consecutive events — such as whether a household's payment amount increased, decreased, or stayed the same across cycles.

sql – payment trend per household
SELECT
  household_id,
  payment_date,
  amount_ngn,
  LAG(amount_ngn) OVER (
    PARTITION BY household_id
    ORDER BY payment_date
  )                                               AS prev_amount,
  amount_ngn - LAG(amount_ngn) OVER (
    PARTITION BY household_id
    ORDER BY payment_date
  )                                               AS change_ngn,
  CASE
    WHEN amount_ngn > LAG(amount_ngn) OVER (
           PARTITION BY household_id ORDER BY payment_date)
    THEN 'increased'
    WHEN amount_ngn < LAG(amount_ngn) OVER (
           PARTITION BY household_id ORDER BY payment_date)
    THEN 'decreased'
    WHEN LAG(amount_ngn) OVER (
           PARTITION BY household_id ORDER BY payment_date) IS NULL
    THEN 'first payment'
    ELSE 'unchanged'
  END                                             AS trend
FROM  nsr.payments
WHERE status = 'processed'
ORDER BY household_id, payment_date;
Reading the output — what each column means
prev_amount: the amount from the previous processed payment for the same household — NULL on the first row because there is no prior payment.

change_ngn: the difference between the current and previous payment. Positive = increase; negative = decrease; NULL = first payment. This is the kind of column a programme analyst uses to flag unusual payment adjustments for audit review.

trend: a human-readable label built with CASE on top of LAG(). The IS NULL branch catches the first payment row correctly — without it, the ELSE 'unchanged' branch would fire for the first row and give a misleading label.

sql – solution
WITH ranked AS (
  -- rank every active household within its state
  SELECT
    h.household_id,
    s.state_name,
    h.pmt_score,
    PERCENT_RANK() OVER (
      PARTITION BY l.state_code
      ORDER BY   h.pmt_score ASC
    ) AS pct_rank
  FROM  nsr.households h
  JOIN  nsr.lgas   l ON h.lga_id     = l.lga_id
  JOIN  nsr.states s ON l.state_code = s.state_code
  WHERE h.is_active = TRUE
),
paid_households AS (
  -- households that have at least one processed payment
  SELECT DISTINCT household_id
  FROM  nsr.payments
  WHERE status = 'processed'
)
SELECT
  r.state_name,
  COUNT(r.household_id)                                 AS total_households,
  COUNT(CASE WHEN r.pct_rank <= 0.20 THEN 1 END)        AS vulnerable_households,
  COUNT(CASE WHEN r.pct_rank <= 0.20
             AND ph.household_id IS NULL THEN 1 END)    AS vulnerable_never_paid,
  COALESCE(SUM(CASE WHEN p.status = 'processed'
               THEN p.amount_ngn END), 0)             AS total_disbursed_ngn,
  ROUND(
    100.0 * COUNT(CASE WHEN p.status = 'failed' THEN 1 END)
           / NULLIF(COUNT(p.payment_id), 0), 1
  )                                                      AS failure_rate_pct
FROM      ranked r
LEFT JOIN paid_households ph ON ph.household_id = r.household_id
LEFT JOIN nsr.payments     p  ON p.household_id  = r.household_id
GROUP BY  r.state_name
ORDER BY  vulnerable_never_paid DESC;
Key decisions in this solution
Two CTEs with distinct responsibilities: ranked handles all window function computation; paid_households pre-aggregates payment eligibility. Run each as a standalone SELECT first to verify before combining.

vulnerable_never_paid uses two CASE conditions: pct_rank <= 0.20 AND ph.household_id IS NULL — both must be true simultaneously. The IS NULL comes from the LEFT JOIN to paid_households finding no match (never paid).

NULLIF prevents division-by-zero: states with no payment records produce COUNT = 0; NULLIF converts that to NULL so the division returns NULL rather than an error.

ORDER BY vulnerable_never_paid DESC: states with the most unserved vulnerable households appear first — this is the priority view for programme intervention.

Basics — Roles, Privileges, and the Principle of Least Privilege

In PostgreSQL, a role is a named identity that can log in and own database objects. A privilege is a permission granted to a role: SELECT, INSERT, UPDATE, DELETE, CONNECT, and so on. Roles do not have access to anything by default — you must explicitly GRANT each privilege. The principle of least privilege means every role gets only what it needs to do its job, and nothing more.

sql – roles from scratch
-- Create a login role with a password
CREATE ROLE alice WITH LOGIN PASSWORD 's3cr3t';

-- Grant connection to a specific database
GRANT CONNECT ON DATABASE lab_nsr TO alice;

-- Grant schema access (required before table access)
GRANT USAGE ON SCHEMA nsr TO alice;

-- Grant table-level privilege
GRANT SELECT ON nsr.households TO alice;

-- Revoke when no longer needed
REVOKE SELECT ON nsr.households FROM alice;

-- Verify privileges on a table
SELECT grantee, privilege_type, is_grantable
FROM  information_schema.role_table_grants
WHERE table_schema = 'nsr'
  AND table_name  = 'households';
GRANT USAGE ON SCHEMA — the step beginners forget
Even if you GRANT SELECT on a table, a role cannot reach it without USAGE on the schema first. The schema is the namespace; the table is inside it. Without USAGE the role gets "schema does not exist" or "permission denied for schema" — a confusing error that looks like the table grant failed. Always grant schema USAGE before table privileges.

Why three separate roles?
The principle of least privilege: each role receives only the permissions it needs and nothing more. nsr_caseworker has no write access and no direct NIN visibility — it reads through a view that masks sensitive columns. nsr_supervisor can UPDATE payments but cannot delete audit records. nsr_dba has BYPASSRLS (needed for maintenance) but even DBA cannot delete from audit_log. Advanced — Column-Level Privileges

Table-level GRANT gives access to all columns. Column-level grants let you expose only specific columns — useful when a table contains mixed-sensitivity data that you cannot fully split into separate tables or views.

sql – column-level privileges
-- Grant SELECT on specific columns only (not NIN, not payment amounts)
GRANT SELECT (household_id, household_code, lga_id, pmt_score, is_active)
  ON nsr.households TO nsr_caseworker;

-- The caseworker can now query only those columns:
SELECT household_code, pmt_score FROM nsr.households; -- ✓ works
SELECT * FROM nsr.households;                       -- ✗ ERROR: permission denied

-- Audit which columns a role has access to
SELECT grantee, column_name, privilege_type
FROM  information_schema.column_privileges
WHERE table_schema = 'nsr'
  AND table_name   = 'households'
ORDER BY grantee, column_name;

-- In practice, a masked VIEW is cleaner than column grants for sensitive data
-- nsr.households has no NIN column (NIN lives in nsr.individuals)
-- We mask head_name here to illustrate the pattern — same technique applies to NIN on individuals
CREATE OR REPLACE VIEW nsr.v_households_safe AS
SELECT
  household_id,
  household_code,
  lga_id,
  pmt_score,
  is_active,
  LEFT(head_name, 1) || REPEAT('*', LENGTH(head_name) - 1) AS head_name_masked
FROM  nsr.households;

-- For NIN masking, use the same pattern on nsr.individuals:
-- LEFT(nin, 3) || '****' || RIGHT(nin, 2) AS nin_partial

GRANT SELECT ON nsr.v_households_safe TO nsr_caseworker;
REVOKE SELECT ON nsr.households FROM nsr_caseworker;  -- revoke direct table access
Views vs column grants — which to use
Column grants are simpler to set up but harder to maintain — adding a new column to the table requires re-issuing the grant. A security view (nsr.v_households_safe) is more flexible: it can apply masking logic, join in lookup labels, and the WITH CHECK OPTION clause prevents writes that bypass the view's filter. For NSR's NIN masking requirement, a view is the right tool because the masking formula can be updated in one place without touching any GRANT statements.

Basics — Why RBAC Alone Is Not Enough

RBAC controls which tables a role can access. But in the NSR, a caseworker granted SELECT on nsr.households can read every household in Nigeria — including states they have no business seeing. Row-Level Security (RLS) adds a second filter: even if the table privilege is granted, each row is checked against a policy before it is returned. RBAC is the door; RLS is the passport check inside.

sql – enabling RLS in three steps
-- Step 1: Enable RLS on the table (does nothing visible until a policy is added)
ALTER TABLE nsr.households ENABLE ROW LEVEL SECURITY;

-- Step 2: With no policy, a non-superuser sees ZERO rows (default-deny)
SET ROLE nsr_caseworker;
SELECT COUNT(*) FROM nsr.households;  -- returns 0
RESET ROLE;

-- Step 3: Create a policy that allows caseworkers to see their own state
-- nsr.households has no state_code column — state is reached via lga_id → nsr.lgas
DROP POLICY IF EXISTS cw_own_state ON nsr.households;
CREATE POLICY cw_own_state
ON nsr.households
FOR SELECT
TO nsr_caseworker
USING (
  lga_id IN (
    SELECT lga_id FROM nsr.lgas
    WHERE  state_code = current_setting('app.state_code', TRUE)
  )
);

-- Now set the session variable and re-test
SET ROLE nsr_caseworker;
SET app.state_code = 'KN';
SELECT COUNT(*) FROM nsr.households;  -- returns Kano rows only
RESET ROLE;
Default-deny is the safe starting point
When RLS is enabled but no policy exists, non-superusers see nothing. This is intentional — it forces explicit decisions about who sees what. A superuser or BYPASSRLS role is never filtered by policies, which is why nsr_dba needs BYPASSRLS to perform maintenance queries across all states.

Intermediate — Row-Level Security

RLS restricts which rows a role can see. Caseworkers should only see households in their assigned state. This is implemented using a session variable app.state_code.

sql – RLS policy
-- Enable RLS on households
ALTER TABLE nsr.households ENABLE ROW LEVEL SECURITY;

-- Policy: caseworkers see only their state
CREATE POLICY caseworker_state_policy
ON nsr.households
FOR SELECT
TO nsr_caseworker
USING (
  lga_id IN (
    SELECT lga_id FROM nsr.lgas
    WHERE state_code = current_setting('app.state_code', TRUE)
  )
);

-- Test as Kano caseworker
SET ROLE nsr_caseworker;
SET app.state_code = 'KN';
SELECT COUNT(*) FROM nsr.households;
-- Returns only Kano households

SET app.state_code = 'LA';
SELECT COUNT(*) FROM nsr.households;
-- Returns only Lagos households

RESET ROLE;
How the RLS policy works
current_setting('app.state_code', TRUE) reads a session-level GUC set by the application or the caseworker's login script. The second argument TRUE means "return NULL if not set" rather than throwing an error. The policy sub-selects LGA IDs belonging to that state, so a Kano caseworker only ever sees rows where lga_id falls within Kano. BYPASSRLS on nsr_dba and nsr_supervisor is intentional — they need cross-state visibility for reports and corrections; the audit log compensates by recording every action they take.

Advanced — USING vs WITH CHECK

A policy has two optional clauses that control different directions of data access. Understanding the difference prevents a subtle security gap: a policy with only USING controls what a caseworker can read, but without WITH CHECK they could still insert or update a row into a state they cannot see.

ClauseApplied toDirectionExample
USING (expr)Existing rowsRead & write targetFilters rows returned by SELECT; also filters which UPDATE/DELETE targets are allowed
WITH CHECK (expr)New/updated row valuesWrite resultValidates the row being INSERTed or the new values after an UPDATE. Prevents writing out-of-state rows.
sql – USING vs WITH CHECK illustrated
-- Policy with USING only: caseworker can read and modify Kano rows,
-- but could INSERT a row with a Lagos lga_id if not checked
CREATE POLICY cw_read_only_state
ON nsr.households FOR ALL TO nsr_caseworker
USING (
  lga_id IN (SELECT lga_id FROM nsr.lgas
              WHERE state_code = current_setting('app.state_code', TRUE))
);
-- Without WITH CHECK, a caseworker SET app.state_code='KN' could still INSERT
-- a row with an lga_id from Lagos — the USING filter only guards reads

-- Correct approach: add WITH CHECK to validate the written row too
DROP POLICY cw_read_only_state ON nsr.households;

CREATE POLICY cw_own_state_full
ON nsr.households FOR ALL TO nsr_caseworker
USING (
  lga_id IN (SELECT lga_id FROM nsr.lgas
              WHERE state_code = current_setting('app.state_code', TRUE))
)
WITH CHECK (
  lga_id IN (SELECT lga_id FROM nsr.lgas
              WHERE state_code = current_setting('app.state_code', TRUE))
);
-- Now: SET app.state_code='KN' then INSERT a Lagos lga_id
-- Expected: ERROR: new row violates row-level security policy

Advanced — Permissive vs Restrictive Policies

By default all policies are permissive — if you have two permissive policies, a row is visible if either passes (they are OR'd together). A restrictive policy is AND'd with everything else — a row must pass the restrictive policy regardless of what other policies allow. Use restrictive policies for mandatory controls that must never be overridden by a more permissive policy.

sql – permissive vs restrictive policies
-- Scenario: caseworkers can see their own state (permissive),
-- BUT active-only records is a mandatory data quality rule (restrictive)

-- Permissive: caseworker sees their state
CREATE POLICY perm_own_state
ON nsr.households AS PERMISSIVE FOR SELECT TO nsr_caseworker
USING (
  lga_id IN (SELECT lga_id FROM nsr.lgas
              WHERE state_code = current_setting('app.state_code', TRUE))
);

-- Restrictive: NOBODY (in policy scope) sees deactivated records
CREATE POLICY restr_active_only
ON nsr.households AS RESTRICTIVE FOR SELECT TO nsr_caseworker
USING (is_active = TRUE);

-- Result: caseworker sees rows where (own state) AND (is_active = TRUE)
-- Even if a future policy relaxed the state check, the active-only rule still applies

-- Test: count all rows vs active rows in Kano
SET ROLE nsr_caseworker;
SET app.state_code = 'KN';
SELECT COUNT(*) FROM nsr.households;          -- Kano active rows only
SELECT COUNT(*) FROM nsr.households
  WHERE is_active = FALSE;                     -- returns 0 — restrictive policy hides them
RESET ROLE;

Advanced — FORCE ROW LEVEL SECURITY

By default, the table owner bypasses RLS even when it is enabled — they can always see all rows. For a government payment system this is a risk: if a caseworker's role is also the schema owner, they bypass the state filter. FORCE ROW LEVEL SECURITY closes this gap by applying policies to the owner as well. BYPASSRLS roles still bypass regardless.

sql – force RLS on table owner
-- Without FORCE, the table owner sees all rows regardless of policies
SET ROLE postgres;  -- postgres is the owner of nsr.households
SELECT COUNT(*) FROM nsr.households;  -- returns ALL rows (policies ignored)
RESET ROLE;

-- Enable FORCE so even the owner is subject to policies
ALTER TABLE nsr.households FORCE ROW LEVEL SECURITY;

-- Now the owner must also satisfy the policy expressions
-- Note: BYPASSRLS roles (nsr_dba, nsr_supervisor) are STILL exempt
-- FORCE only affects the table owner when they are not BYPASSRLS

-- To undo:
ALTER TABLE nsr.households NO FORCE ROW LEVEL SECURITY;

-- Summary view: RLS status on all NSR tables
SELECT relname AS table_name,
       relrowsecurity  AS rls_enabled,
       relforcerowsecurity AS force_rls
FROM  pg_class c
JOIN  pg_namespace n ON n.oid = c.relnamespace
WHERE n.nspname = 'nsr'
  AND c.relkind = 'r'  -- 'r' = regular table
ORDER BY relname;
RLS is not a substitute for application-level validation
RLS stops unauthorised data access at the database layer — it is a last line of defence. Your application should still validate state_code on login, enforce it in all queries, and never expose raw database errors to end users. Think of RLS as the database refusing to lie down even if the application misbehaves.

Basics — What Audit Logging Is and Why Databases Need It

An audit log answers the question: "Who changed what, when, and from where?" Application logs can be disabled or cleared; network logs don't capture the data values; only a database-level audit captures the exact row changed, the role that changed it, and the timestamp — inside the same ACID-protected system as the data itself.

Why the NSR uses trigger-based logging rather than relying on pgAudit alone
pgAudit records that a statement ran and who ran it — it writes to the PostgreSQL server log file, which is not queryable by SQL and is not stored in the database itself. For the NSR, investigators need to answer "show me every change to household HH-LA-001 over the last six months" — that requires row-level before/after values stored in a table, not grep-able log files. pgAudit is a complement (session-level DDL monitoring); the trigger-based audit_log is the primary forensic tool.

Why the audit log is append-only

Intermediate — The Audit Trigger Function

The audit_log table is only useful if records are written to it automatically on every INSERT, UPDATE, and DELETE. A PostgreSQL trigger function does this — it fires after every row change on the target table, captures NEW and OLD row data, and inserts a record into audit_log. The function uses SECURITY DEFINER so it can write to audit_log even when the calling role (e.g. nsr_caseworker) has no INSERT privilege on it.

sql – audit trigger function
-- Step 1: Create the trigger function
-- RETURNS TRIGGER is required for all trigger functions
-- SECURITY DEFINER means it runs as its owner (postgres), not the calling role
CREATE OR REPLACE FUNCTION nsr.fn_audit_log()
RETURNS TRIGGER
LANGUAGE plpgsql
SECURITY DEFINER
AS $$
DECLARE
  v_row_data JSONB;
BEGIN
  -- TG_OP is 'INSERT', 'UPDATE', or 'DELETE'
  -- NEW holds the row after the change; OLD holds it before
  IF TG_OP = 'DELETE' THEN
    v_row_data := row_to_json(OLD)::JSONB;  -- capture the deleted row
  ELSIF TG_OP = 'INSERT' THEN
    v_row_data := row_to_json(NEW)::JSONB;  -- capture the new row
  ELSE  -- UPDATE: store the NEW row; OLD is available if you need before/after
    v_row_data := jsonb_build_object(
      'old', row_to_json(OLD)::JSONB,
      'new', row_to_json(NEW)::JSONB
    );
  END IF;

  -- Always pass session_user explicitly — do NOT rely on the column DEFAULT
  -- Inside SECURITY DEFINER, current_user = function owner (postgres), not the caller
  -- session_user always holds the original authenticated role
  INSERT INTO nsr.audit_log (role_name, table_name, action, row_data)
  VALUES (session_user, TG_TABLE_SCHEMA || '.' || TG_TABLE_NAME, TG_OP, v_row_data);

  -- AFTER triggers: return value is ignored by PostgreSQL
  -- Convention: return NEW for INSERT/UPDATE, OLD for DELETE
  RETURN COALESCE(NEW, OLD);
END;
$$;

-- Step 2: Attach the trigger to nsr.households
-- DROP IF EXISTS makes this block safe to re-run
DROP TRIGGER IF EXISTS trg_audit_households ON nsr.households;
CREATE TRIGGER trg_audit_households
AFTER INSERT OR UPDATE OR DELETE ON nsr.households
FOR EACH ROW EXECUTE FUNCTION nsr.fn_audit_log();

-- Step 3: Test it — insert a household, then check the audit log
-- head_name is NOT NULL — always include it (check lga_id with: SELECT lga_id, lga_name FROM nsr.lgas LIMIT 5)

VALUES ('HH-TEST-999', 2, 'Test Household', 30.0, TRUE);

FROM  nsr.audit_log
WHERE table_name = 'nsr.households'
ORDER BY log_time DESC LIMIT 3;
-- You should see the INSERT with row_data containing the full new row as JSONB

-- Step 4: Update a pmt_score and confirm before/after is captured

WHERE  household_code = 'HH-TEST-999';

       row_data->'old'->>'pmt_score' AS score_before,
       row_data->'new'->>'pmt_score' AS score_after
FROM  nsr.audit_log
WHERE table_name = 'nsr.households'
  AND action = 'UPDATE'
ORDER BY log_time DESC LIMIT 1;

A well-designed audit_log table is queryable with standard SQL. These four patterns cover the most common forensic scenarios an NSR DBA or supervisor faces in practice — from tracing a single household's history to detecting a potential fraud signal.

sql – forensic audit queries
-- Query 1: Full change history of one household
-- Use this when a supervisor asks "who changed HH-LA-001 and when?"
-- COALESCE handles two row_data shapes:
--   INSERT/DELETE → flat:   {"household_code":"HH-LA-001","pmt_score":"28.5",...}
--   UPDATE        → nested: {"old":{...},"new":{...}}
-- Without COALESCE, ->> returns NULL for every UPDATE row — silently wrong
SELECT log_time, role_name, action,
       COALESCE(row_data->>'pmt_score',
                row_data->'new'->>'pmt_score')      AS pmt_score,
       COALESCE(row_data->>'household_code',
                row_data->'new'->>'household_code') AS code
FROM  nsr.audit_log
WHERE table_name = 'nsr.households'
  AND COALESCE(row_data->>'household_code',
              row_data->'new'->>'household_code') = 'HH-LA-001'
ORDER BY log_time;

-- Query 2: All changes by a specific role in the last 7 days
-- Use this when investigating a caseworker's activity
SELECT log_time, table_name, action, row_data
FROM  nsr.audit_log
WHERE role_name = 'nsr_caseworker'
  AND log_time >= now() - INTERVAL '7 days'
ORDER BY log_time DESC;

-- Query 3: Detect bulk changes — potential fraud signal
-- Flags any role that made more than 100 UPDATE/DELETE in one hour
SELECT role_name,
       DATE_TRUNC('hour', log_time) AS hour_bucket,
       COUNT(*) AS changes
FROM  nsr.audit_log
WHERE action IN ('UPDATE', 'DELETE')
  AND log_time >= now() - INTERVAL '24 hours'
GROUP BY role_name, hour_bucket
HAVING COUNT(*) > 100
ORDER BY changes DESC;
-- With only a few test rows this returns nothing — that is correct (no fraud signal)

-- Query 4: Before/after comparison — explicit CASE on row_data shape
-- LAG() does not work here: UPDATE rows store {"old":...,"new":...} so
-- row_data->>'pmt_score' returns NULL, and LAG() just chains NULLs.
-- The trigger already captures old and new in one row for UPDATE — use that directly.
SELECT
  log_time,
  role_name,
  action,
  CASE action
    WHEN 'INSERT' THEN NULL
    WHEN 'DELETE' THEN row_data->>'pmt_score'
    WHEN 'UPDATE' THEN row_data->'old'->>'pmt_score'
  END AS score_before,
  CASE action
    WHEN 'INSERT' THEN row_data->>'pmt_score'
    WHEN 'DELETE' THEN NULL
    WHEN 'UPDATE' THEN row_data->'new'->>'pmt_score'
  END AS score_after
FROM  nsr.audit_log
WHERE table_name = 'nsr.households'
  AND COALESCE(row_data->>'household_code',
              row_data->'new'->>'household_code') = 'HH-LA-001'
ORDER BY log_time;
-- INSERT row: score_before = NULL, score_after = the initial value
-- UPDATE row: score_before and score_after both populated from old/new
-- DELETE row: score_before = the last value, score_after = NULL
JSONB operators used here — and when to use an index
->> extracts a JSONB field as text — right for display and equality checks (= 'HH-LA-001'). On a small audit log this is fast enough. On a production NSR log with millions of rows, add a GIN index: CREATE INDEX ON nsr.audit_log USING GIN (row_data) — then use the containment operator @> instead: WHERE row_data @> '{"household_code":"HH-LA-001"}'::jsonb. The GIN index makes that containment check O(log N) rather than scanning every row.

Core Terms — Backup & Disaster Recovery at a Glance

TermWhat it means
Logical backupA human-readable export of database objects and data as SQL statements or a binary archive. Created with pg_dump. The database must be running. Cannot capture in-progress transactions.
Physical backupA byte-for-byte copy of the PostgreSQL data directory files. Created with pg_basebackup. Works independently of the database's logical structure and supports Point-in-Time Recovery when combined with WAL archiving.
WAL (Write-Ahead Log)PostgreSQL's internal change journal. Every INSERT, UPDATE, and DELETE is written to WAL before the data files are updated. WAL files are the foundation of replication and PITR — they can be replayed to rebuild the database to any past state.
WAL archivingContinuously copying WAL segment files to a safe location (another server, S3, NFS). Enables recovery to any point in time between backups, not just the backup moment.
PITR (Point-in-Time Recovery)Restoring a physical base backup then replaying archived WAL up to a specific timestamp. Allows recovery from accidental deletions or corruption with near-zero data loss.
RPO (Recovery Point Objective)The maximum acceptable amount of data loss measured in time. A daily pg_dump has 24-hour RPO. WAL archiving achieves near-zero RPO (seconds of data loss).
RTO (Recovery Time Objective)The maximum acceptable time the system can be offline during recovery. A logical restore of a large database can take hours; a physical restore with a warm standby can be minutes.
pg_dumpPostgreSQL's logical backup tool. Can export an entire database, a single schema, or a single table. Supports plain SQL (-F p), custom binary (-F c), directory (-F d), and tar (-F t) formats.
pg_restoreRestores a non-plain-text pg_dump archive. Supports parallel restore with -j N to use N CPU cores simultaneously — dramatically faster for large databases.
pg_basebackupCreates a physical backup of the running PostgreSQL cluster, including WAL. The starting point for PITR.
Streaming replicationA standby server that continuously receives WAL from the primary and stays in sync. A warm standby can be promoted to primary within seconds if the primary fails.

Basics — Backup Types and Why They Differ

PostgreSQL offers two fundamental backup approaches. Understanding the difference determines how quickly you can recover and how much data you risk losing in a disaster.

TypeHow it worksRecovery granularityNSR use case
Logical backup (pg_dump)Exports data as SQL statements or binary; database must be accessiblePoint-in-time only at backup momentSchema migration, developer copies, schema-only restores
Physical backup (pg_basebackup)Copies the raw data files + WAL; works at file levelAny point in time with WAL archiving (PITR)Production DR, standby servers, zero data loss recovery
Recovery Point Objective (RPO) and Recovery Time Objective (RTO)
RPO: how much data can you afford to lose? A daily pg_dump gives you 24-hour RPO — if the server fails at 11 pm, you lose a day of changes. WAL archiving gives you near-zero RPO — you can recover to within seconds of the failure. RTO: how long can the system be unavailable? A logical restore of a large NSR database can take hours; a physical restore with a warm standby can be minutes. For a government payment system, both numbers need to be defined in an SLA and tested regularly — not estimated.

Intermediate — Backup Verification: the Anti-Pattern to Avoid

The most dangerous backup mistake is never testing the restore. A backup that has never been restored successfully is not a backup — it is a false sense of security. For the NSR, an untested backup could mean weeks of lost payment records in a real incident.

The Golden Rule of Backups
A backup is only as good as the last successful restore test. Schedule restore drills on a test database at least monthly. Verify: (1) row counts match, (2) sequences are not reset, (3) all GRANT statements were preserved, (4) RLS policies are in place on the restored database.

Advanced — WAL Archiving and Point-in-Time Recovery (PITR)

Logical backups capture a snapshot in time. WAL archiving continuously ships WAL segment files to a safe location so you can recover to any point between backups — not just the moment the backup was taken. This is essential for a payment system where even one hour of lost transactions is unacceptable.

PITR in a real NSR incident — what recovery looks like
Scenario: at 14:47 a caseworker accidentally runs DELETE FROM nsr.payments WHERE state_code = 'KN' without a WHERE clause refinement — 12,000 payment records deleted, transaction committed. With pg_dump only, you restore to last night's backup and lose a day of legitimate entries. With WAL archiving, you restore the base backup, apply WAL up to 14:46:59, and lose nothing. The key steps: stop the corrupted database, restore the base backup files, set recovery_target_time to just before the bad transaction, start in recovery mode. PostgreSQL replays all WAL up to that timestamp and stops — the bad DELETE never happened. Then promote the instance to accept connections. This is the DR procedure every participant should be able to explain and execute.

Why both USING and WITH CHECK are required
USING filters which existing rows a role can read or target. It does not validate the values being written. Without WITH CHECK, a Kano caseworker can INSERT a row with a Lagos lga_id — it succeeds because the check on reads is irrelevant to a write. WITH CHECK validates the new row before it is stored. Use the same predicate in both clauses. FOR SELECT only policies have no write risk but miss DELETE and UPDATE row-targeting — always use FOR ALL unless you have a deliberate reason to scope by command.

  log_id     BIGSERIAL   PRIMARY KEY,
  log_time   TIMESTAMPTZ NOT NULL DEFAULT now(),
  role_name  TEXT        NOT NULL DEFAULT session_user,
  table_name TEXT        NOT NULL,
  action     TEXT        NOT NULL,
  row_data   JSONB
);
REVOKE DELETE ON nsr.audit_log FROM PUBLIC;
REVOKE DELETE ON nsr.audit_log FROM nsr_dba;

-- Step 2: Trigger function — matches nsr.audit_log schema exactly
-- session_user captures the real caller even inside a SECURITY DEFINER function
CREATE OR REPLACE FUNCTION nsr.audit_payments()
RETURNS TRIGGER LANGUAGE plpgsql SECURITY DEFINER
AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    INSERT INTO nsr.audit_log (table_name, action, row_data)
    VALUES (TG_TABLE_SCHEMA || '.' || TG_TABLE_NAME,
            TG_OP,
            row_to_json(NEW)::JSONB);
    RETURN NEW;

  ELSIF TG_OP = 'UPDATE' THEN
    INSERT INTO nsr.audit_log (table_name, action, row_data)
    VALUES (TG_TABLE_SCHEMA || '.' || TG_TABLE_NAME,
            TG_OP,
            jsonb_build_object('old', row_to_json(OLD)::JSONB,
                               'new', row_to_json(NEW)::JSONB));
    RETURN NEW;

  ELSE -- DELETE
    INSERT INTO nsr.audit_log (table_name, action, row_data)
    VALUES (TG_TABLE_SCHEMA || '.' || TG_TABLE_NAME,
            TG_OP,
            row_to_json(OLD)::JSONB);
    RETURN OLD;
  END IF;
END;
$$;

-- Step 3: Attach to payments table
DROP TRIGGER IF EXISTS trg_audit_payments ON nsr.payments;
CREATE TRIGGER trg_audit_payments
AFTER INSERT OR UPDATE OR DELETE ON nsr.payments
FOR EACH ROW EXECUTE FUNCTION nsr.audit_payments();

Key teaching points
SECURITY DEFINER: the function runs with the privileges of its owner (postgres), not the caller. This lets a restricted caseworker write to audit_log without having a direct INSERT grant on it. session_user vs current_user: inside a SECURITY DEFINER function current_user returns the function owner — always use session_user (the DEFAULT on role_name) to record who actually triggered the change. TG_OP / TG_TABLE_NAME / TG_TABLE_SCHEMA: trigger variables — no hard-coding needed. row_to_json(OLD/NEW)::JSONB: captures the entire row as JSONB — any future column addition is automatically included. RETURN OLD for DELETE: AFTER triggers ignore the return value but convention is to return OLD on DELETE and NEW on INSERT/UPDATE.

WITH CHECK, restrictive policies, and FORCE RLS
Advanced ▼

This exercise exposes the limits of a USING-only policy and demonstrates the full defensive RLS stack: USING + WITH CHECK, a restrictive policy, and FORCE ROW LEVEL SECURITY.

  1. Expose the gap: Create a policy on nsr.households for nsr_caseworker using USING only (state filter). Then as the caseworker, try to INSERT a row with a Lagos lga_id while app.state_code = 'KN'. Does it fail?

Complete steps 1 and 2 in order — if you create WITH CHECK in step 1 you will not observe the gap. The point of step 1 is to prove the vulnerability exists before you fix it.

1A — Enhanced Schema & Seed Data (partitioned disbursements)


    

      
Key difference from a plain CREATE TABLE

      The cwr.disbursements table uses PARTITION BY RANGE (disbursement_date). PostgreSQL requires the partition key to be part of the PRIMARY KEY — so the PK is (disbursement_id, disbursement_date), not just disbursement_id. Every INSERT is automatically routed to the correct annual child table. The parent table itself holds no rows.

1B — Materialized View & Performance Indexes (powers the dashboard)


    

      
Why a materialized view instead of a regular view?

      A regular view runs its query on every SELECT — with six JOINs across large welfare tables this becomes expensive. A materialized view stores the result on disk and is refreshed on demand. The dashboard reads from cwr.mv_zone_stats in microseconds instead of scanning all base tables. Run REFRESH MATERIALIZED VIEW CONCURRENTLY cwr.mv_zone_stats; after any bulk data change.
    


    

      
sql – run as superuser in lab_nsr

      
-- ─────────────────────────────────────────────────────────────────
-- CWR 1B · Materialized View and Composite Indexes
-- ─────────────────────────────────────────────────────────────────

-- Composite indexes for common query patterns in the security layer
CREATE INDEX idx_beneficiaries_community_active
    ON cwr.beneficiaries (community_id, is_active)
    WHERE is_active = TRUE;

CREATE INDEX idx_disbursements_date_status
    ON cwr.disbursements (disbursement_date DESC, status);

CREATE INDEX idx_audit_log_time_table
    ON cwr.audit_log (log_time DESC, table_name);

-- Materialized view: pre-aggregated zone statistics
-- This is what the dashboard reads — 6 JOINs computed once, cached on disk
CREATE MATERIALIZED VIEW cwr.mv_zone_stats AS
SELECT
    z.zone_code,
    z.zone_name,
    COUNT(DISTINCT b.beneficiary_id)                                        AS total_beneficiaries,
    COUNT(DISTINCT b.beneficiary_id) FILTER (WHERE b.is_active)            AS active_beneficiaries,
    ROUND(AVG(b.welfare_score), 2)                                          AS avg_welfare_score,
    COALESCE(SUM(d.amount_ngn) FILTER (WHERE d.status = 'processed'), 0) AS total_disbursed,
    COUNT(d.disbursement_id)     FILTER (WHERE d.status = 'pending')     AS pending_disbursements,
    COUNT(d.disbursement_id)     FILTER (WHERE d.status = 'failed')      AS failed_disbursements
FROM  cwr.zones z
LEFT  JOIN cwr.communities   c ON c.zone_id        = z.zone_id
LEFT  JOIN cwr.beneficiaries b ON b.community_id   = c.community_id
LEFT  JOIN cwr.disbursements d ON d.beneficiary_id = b.beneficiary_id
GROUP  BY z.zone_id, z.zone_code, z.zone_name
ORDER  BY z.zone_code;

CREATE UNIQUE INDEX idx_mv_zone_stats_code ON cwr.mv_zone_stats (zone_code);

-- Initial population — must run before the dashboard can read data
REFRESH MATERIALIZED VIEW cwr.mv_zone_stats;

1C — Monitoring Infrastructure (dashboard health panel)


    

      
Why wrap pg_stat_* in views?

      Application roles should never query system catalogs directly — the schema changes between PostgreSQL versions and the privilege model is complex. By wrapping pg_stat_activity, pg_locks, and pg_stat_bgwriter in schema views, you control exactly which columns are exposed, and you can update the view definition without changing application code.

CREATE OR REPLACE VIEW cwr.v_active_sessions AS
SELECT
    pid,
    usename           AS role_name,
    application_name,
    state,
    wait_event_type,
    wait_event,
    query_start,
    now() - query_start  AS duration,
    LEFT(query, 100)    AS query_snippet
FROM  pg_stat_activity
WHERE datname = current_database()
  AND  pid <> pg_backend_pid()
ORDER  BY query_start;

-- View: lock contention (System Health panel — Lock Contention card)
CREATE OR REPLACE VIEW cwr.v_lock_monitor AS
SELECT
    blocked.pid              AS blocked_pid,
    blocked.usename          AS blocked_user,
    blocking.pid             AS blocking_pid,
    blocking.usename         AS blocking_user,
    LEFT(blocked.query, 80) AS blocked_query,
    now() - blocked.query_start  AS wait_duration
FROM  pg_stat_activity  blocked
JOIN  pg_stat_activity  blocking
      ON  blocking.pid = ANY(pg_blocking_pids(blocked.pid))
WHERE cardinality(pg_blocking_pids(blocked.pid)) > 0;

    ELSIF TG_OP = 'INSERT' THEN
        v_row_data := row_to_json(NEW)::JSONB;
    ELSE
        v_row_data := jsonb_build_object(
            'old', row_to_json(OLD)::JSONB,
            'new', row_to_json(NEW)::JSONB
        );
    END IF;

    INSERT INTO cwr.audit_log (role_name, table_name, action, row_data)
    VALUES (
        session_user,
        TG_TABLE_SCHEMA || '.' || TG_TABLE_NAME,
        TG_OP,
        v_row_data
    );

    RETURN COALESCE(NEW, OLD);
END;
$func$;

DROP TRIGGER IF EXISTS trg_audit_beneficiaries ON cwr.beneficiaries;
CREATE TRIGGER trg_audit_beneficiaries

Pre-built: Forensic Beneficiary Timeline Function (Advanced)


    

      
What this function does

      After Puzzle C is active, every change to cwr.beneficiaries is stored as a JSONB blob in cwr.audit_log. This forensic function unpacks that blob row by row — one output row per changed field — so you can reconstruct the exact before and after values for any individual field change. It uses two advanced patterns: CROSS JOIN LATERAL to expand JSONB keys dynamically, and the same COALESCE shape-detection trick from Day 3 Query 4.
    


    

      
sql – run as superuser in lab_nsr

      
-- ─────────────────────────────────────────────────────────────────
-- CWR 3A · Forensic Beneficiary Timeline — Advanced · Provided
-- ─────────────────────────────────────────────────────────────────

CREATE OR REPLACE FUNCTION cwr.fn_beneficiary_timeline(p_beneficiary_code TEXT)
RETURNS TABLE (
    event_time   TIMESTAMPTZ,
    role_name    TEXT,
    action       TEXT,
    field        TEXT,
    old_value    TEXT,
    new_value    TEXT
)
LANGUAGE sql STABLE SECURITY DEFINER AS $$
    SELECT
        al.log_time,
        al.role_name,
        al.action,
        k.key                                               AS field,
        CASE al.action
            WHEN 'DELETE' THEN al.row_data          ->> k.key
            WHEN 'INSERT' THEN NULL
            WHEN 'UPDATE' THEN al.row_data -> 'old' ->> k.key
        END                                                AS old_value,
        CASE al.action
            WHEN 'DELETE' THEN NULL
            WHEN 'INSERT' THEN al.row_data          ->> k.key
            WHEN 'UPDATE' THEN al.row_data -> 'new' ->> k.key
        END                                                AS new_value
    FROM  cwr.audit_log al
    CROSS JOIN LATERAL (
        SELECT key
        FROM   jsonb_each_text(
            CASE al.action
                WHEN 'UPDATE' THEN al.row_data -> 'new'
                ELSE               al.row_data
            END
        )
    ) k(key)
    WHERE al.table_name = 'cwr.beneficiaries'
      AND   COALESCE(
                al.row_data          ->> 'beneficiary_code',
                al.row_data -> 'new' ->> 'beneficiary_code'
            ) = p_beneficiary_code
    ORDER  BY al.log_time, k.key;
$$;

Basics — What EXPLAIN Does and Why You Need It

      
When you run a SQL query, PostgreSQL's query planner chooses how to execute it — which indexes to use, in what order to join tables, whether to sort or hash. EXPLAIN shows you that plan without executing the query. EXPLAIN ANALYZE actually runs the query and shows both the plan and real measured timings, so you can see where the time actually went.

      

        
sql – three levels of EXPLAIN

        
-- Level 1: estimated plan only (no execution)
EXPLAIN SELECT * FROM nsr.households WHERE pmt_score < 30;

-- Level 2: real timings (runs the query)
EXPLAIN ANALYZE SELECT * FROM nsr.households WHERE pmt_score < 30;

-- Level 3: real timings + buffer (cache hit vs disk) breakdown
EXPLAIN (ANALYZE, BUFFERS) SELECT * FROM nsr.households WHERE pmt_score < 30;

      

      

        
How to read the plan output — three numbers that matter

        Every plan node shows: (cost=start..total rows=N width=B). cost: a dimensionless estimate (not milliseconds) — lower is better, but the number only matters relative to other nodes. rows: how many rows the planner thinks this node will return — if this is wildly wrong vs the actual rows in ANALYZE output, your statistics are stale and the planner is choosing a bad plan. actual time=X..Y: X ms to return the first row, Y ms to return all rows. The node with the highest actual time is your bottleneck. Work from the innermost node outward.
      


      
Intermediate — Reading an Execution Plan

      

        
sql

        
-- Always use ANALYZE to get real timings
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT)
SELECT h.household_code, h.pmt_score, s.state_name
FROM nsr.households h
JOIN nsr.lgas   l ON h.lga_id     = l.lga_id
JOIN nsr.states s ON l.state_code = s.state_code

How to read the output

        The plan is read bottom-up: the innermost (most indented) node executes first. Key numbers to compare: rows (planner estimate) vs actual rows (real execution) — a large gap means stale statistics; run ANALYZE nsr.households; to fix. cost is an arbitrary planner unit (not milliseconds); only compare costs within the same plan. actual time is real milliseconds for that node. Buffers: shared hit = pages served from shared_buffers (fast); shared read = pages fetched from disk (slow). A query with high shared reads on a table that should be cached is a sign either shared_buffers is too small or the table has severe bloat from dead tuples.
      

    


    

      
Basics — What an Index Is and How It Works

      
An index is a separate data structure that PostgreSQL maintains alongside a table, mapping column values to the physical locations of matching rows. Without an index, every query must read every row in the table (a sequential scan). With an index, PostgreSQL jumps directly to the relevant rows — like using a book's index rather than reading every page.

      

        
Without indexWith B-tree index

        
Scan all N rows, test each oneBinary-search the index tree — O(log N) lookups

        
Fast for full-table reads (seq scan)Fast for selective queries (<~10% of rows)

        
No storage overheadExtra disk space + maintenance on every INSERT/UPDATE/DELETE

      

      

        
sql – create and verify a basic index

        
-- Without an index: Seq Scan on large table
EXPLAIN ANALYZE SELECT * FROM nsr.households WHERE pmt_score < 30;

-- Create a B-tree index on pmt_score
CREATE INDEX idx_hh_pmt_score ON nsr.households (pmt_score);

-- With index: Index Scan or Bitmap Heap Scan
EXPLAIN ANALYZE SELECT * FROM nsr.households WHERE pmt_score < 30;

-- Confirm the index exists and is being used
SELECT indexrelname, idx_scan, idx_tup_read
FROM  pg_stat_user_indexes
WHERE relname = 'households';

      

      

        
When PostgreSQL ignores your index

        The planner chooses a sequential scan when it estimates the index would be slower — this happens when the query is not selective enough (e.g. pmt_score < 100 returns 80% of rows — faster to scan the whole table). It also happens when statistics are stale (ANALYZE fixes this) or when the column is wrapped in a function (WHERE UPPER(lga_name) = 'KANO' cannot use a regular index — you need a functional index: CREATE INDEX ON nsr.lgas (UPPER(lga_name))).
      


      
Intermediate — Index Types and When to Use Them

      

        
sql – index creation

        
-- B-tree (default): equality and range queries
CREATE INDEX idx_households_pmt_score
ON nsr.households (pmt_score);

-- Partial index: only index active households (much smaller)
CREATE INDEX idx_households_active_pmt
ON nsr.households (pmt_score)
WHERE is_active = TRUE;

-- Composite index: state + score (covers the ORDER BY in common queries)
CREATE INDEX idx_lgas_state_lga
ON nsr.lgas (state_code, lga_name);

-- GIN index: JSONB columns
CREATE INDEX idx_audit_row_data
ON nsr.audit_log USING GIN (row_data);

-- Check index usage
SELECT indexrelname, idx_scan, idx_tup_read
FROM pg_stat_user_indexes
WHERE schemaname = 'nsr'
ORDER BY idx_scan DESC;

      

      

        
Choosing the right index type

        B-tree (default): handles equality, range, BETWEEN, ORDER BY. Right for pmt_score, payment_date, household_code. Partial index: adds a WHERE clause to the index definition — only rows satisfying the condition are indexed. Smaller, faster, used when the common query always filters on the same condition (e.g. is_active = TRUE). Composite index: covers multi-column WHERE and ORDER BY; column order matters — leading column must appear in the query for the index to be used. GIN: required for JSONB containment (@>) and full-text search — cannot be used for range queries. The pg_stat_user_indexes query at the end is the index audit: any index with idx_scan = 0 after weeks of production use is unused overhead on every INSERT/UPDATE and should be dropped.
      

    


    

      
Basics — What Partitioning Is and When You Need It

      
Partitioning splits one logical table into multiple physical storage files called partitions, based on a column's value. To users and applications, it still looks like one table — but PostgreSQL can skip entire partitions when they cannot possibly contain matching rows, dramatically reducing I/O on large tables.

      

        
Problem without partitioningSolution with partitioning

        
Query for 2024 payments scans all yearsPostgreSQL reads only the 2024 partition

        
VACUUM must process entire tableVACUUM can target one partition at a time

        
Old data slows every queryDrop old partition instantly with no table lock

        
One large index for all yearsSmaller per-partition indexes fit in cache

      

      

        
When partitioning is worth it — and when it is not

        Partitioning pays off when: the table has tens of millions of rows; most queries filter on the partition key; you need fast bulk deletes (drop the partition). It adds complexity and is not worth it for tables under ~5 million rows or when queries do not filter on the partition key. For the NSR payments table — growing at millions of rows per payment cycle across 36 states — partitioning by payment_date year is a clear win. For the states reference table (37 rows), partitioning would be absurd.
      


      
Intermediate — Table Partitioning

      
Partition large tables by a column that is commonly used in WHERE clauses — for the NSR, payments by year or households by state.

      

        
sql – range partitioning by year

        
-- Create partitioned parent
CREATE TABLE nsr.payments_partitioned (
  payment_id    BIGSERIAL,
  household_id  BIGINT        NOT NULL,
  payment_date  DATE          NOT NULL,
  amount_ngn    NUMERIC(12,2),
  status        TEXT
) PARTITION BY RANGE (payment_date);

-- Create partitions
CREATE TABLE nsr.payments_2024
  PARTITION OF nsr.payments_partitioned
  FOR VALUES FROM ('2024-01-01') TO ('2025-01-01');

CREATE TABLE nsr.payments_2025
  PARTITION OF nsr.payments_partitioned
  FOR VALUES FROM ('2025-01-01') TO ('2026-01-01');

EXPLAIN SELECT * FROM nsr.payments_partitioned
WHERE payment_date BETWEEN '2024-03-01' AND '2024-06-30';
-- Only payments_2024 appears in the plan

      

      

        
Why partition payments by year?

        The NSR payments table will accumulate millions of rows across multiple disbursement cycles. A caseworker reporting on 2024 payments should never need to scan 2025 or 2026 data. Partition pruning lets PostgreSQL eliminate irrelevant child tables entirely — the EXPLAIN output shows only payments_2024 in the plan when filtering by a 2024 date range, regardless of how large the other partitions grow. Each partition can also have its own index, vacuum schedule, and tablespace — enabling hot/cold storage separation: recent partitions on fast NVMe, archived partitions on cheaper storage. The key constraint: the partition key column (payment_date) must always appear in queries for pruning to work; queries without a date filter scan all partitions and may be slower than a non-partitioned table.
      

    


    

      
Basics — Why Database Monitoring Matters

      
A database that runs well today can degrade silently over weeks — dead tuples accumulate, indexes go unused, connections pile up, and cache hit rates drop. Monitoring means running a small set of SQL queries against PostgreSQL's built-in statistics views to catch these problems before users notice them. All monitoring data lives in pg_stat_* views and is updated continuously by the statistics collector.

Basics — What ETL Is and Why Data Migration Is Hard

      
ETL (Extract, Transform, Load) is the process of moving data from one system into another while cleaning and reshaping it along the way. It sounds simple but is one of the most failure-prone operations in database work — because source data is almost always dirtier than expected. The NSR receives household data from 774 LGAs collected by field enumerators using paper forms and mobile apps under poor connectivity. Expect nulls, duplicates, invalid codes, mismatched formats, and truncated names — always.

      

        
sql – the simplest possible ETL: load a CSV into PostgreSQL

        
-- Step 1: Create a staging table with NO constraints (accept everything)
CREATE TABLE nsr.stg_raw (
  raw_line  TEXT   -- accept as plain text first; parse later
);

-- Step 2: Load the file (run in psql, not in SQL)
-- \COPY nsr.stg_raw FROM '/tmp/lga_data.csv' CSV HEADER

-- Step 3: Inspect what arrived
SELECT COUNT(*) FROM nsr.stg_raw;
SELECT * FROM  nsr.stg_raw LIMIT 5;

-- Step 4: Count problems before deciding how to fix them
SELECT
  COUNT(*) FILTER (WHERE raw_line IS NULL)          AS null_rows,
  COUNT(*) FILTER (WHERE LENGTH(raw_line) < 5)        AS suspiciously_short,
  COUNT(*)                                             AS total_rows
FROM  nsr.stg_raw;

      

      

        
The cardinal rule of ETL: never transform in place

        Always load raw data into a staging table first, validate it there, and only promote clean rows to the target table. If you transform as you load — casting types, applying constraints, resolving FKs — a single bad row aborts the entire COPY and you get nothing. With a staging table, you get everything, then make deliberate decisions about what to keep and what to quarantine.

PostgreSQL-Native ETL Pattern

      

        
sql – staging → target with validation

        
-- Step 1: Create staging table (no constraints)
CREATE TABLE nsr.stg_households (
  household_code  TEXT,
  state_name      TEXT,
  lga_name        TEXT,
  head_name       TEXT,
  pmt_score       TEXT,
  import_batch    TEXT
);

-- Step 2: Load from CSV
-- \COPY nsr.stg_households FROM '/tmp/nsr_raw.csv' CSV HEADER

-- Step 3: Validate
SELECT 'missing_code' AS issue, COUNT(*) FROM nsr.stg_households WHERE household_code IS NULL
UNION ALL
SELECT 'invalid_pmt',   COUNT(*) FROM nsr.stg_households WHERE pmt_score !~ '^[0-9]+\.?[0-9]*$'
UNION ALL
SELECT 'duplicate_code', COUNT(*) - COUNT(DISTINCT household_code)
                          FROM nsr.stg_households;

-- Step 4: Load valid rows (with type cast)
INSERT INTO nsr.households (household_code, lga_id, head_name, pmt_score)
SELECT
  s.household_code,
  l.lga_id,
  s.head_name,
  s.pmt_score::NUMERIC
FROM nsr.stg_households s
JOIN nsr.lgas l ON l.lga_name = s.lga_name
WHERE s.household_code IS NOT NULL
  AND s.pmt_score ~ '^[0-9]+\.?[0-9]*$'
ON CONFLICT (household_code) DO NOTHING;

      

      

        
Why staging tables have no constraints

        The staging table accepts everything — nulls, bad formats, duplicates — because field enumeration data from Nigeria's LGAs will contain all of these. Rejecting bad data at load time means losing context about what arrived. Instead: load everything, validate in SQL, quarantine or fix bad rows, then promote clean rows into the target. pmt_score as TEXT in staging is deliberate — the regex '^[0-9]+\.?[0-9]*$' in the validation step catches values like "N/A" or "–" that would silently convert to NULL or cause a type error if the column were NUMERIC. ON CONFLICT DO NOTHING makes the INSERT idempotent — re-running after a failure does not create duplicates. In a re-enumeration scenario, use ON CONFLICT DO UPDATE instead to refresh existing records from the new batch.
      

    


    

      
Basics — Why Duplicates Exist and Why They Are Dangerous

      
Duplicates enter databases in two main ways: technical duplicates (the same row loaded twice due to a failed ETL re-run) and data duplicates (the same real-world entity recorded under different IDs, e.g. a household enumerated twice by different enumerators). The consequences are different: technical duplicates inflate counts and double payments; data duplicates require human investigation because you cannot safely auto-merge without risking data loss.

      

        
sql – detecting duplicates at the basics level

        
-- How many distinct household codes do we have?
SELECT COUNT(*) AS total_rows,
       COUNT(DISTINCT household_code) AS unique_codes,
       COUNT(*) - COUNT(DISTINCT household_code) AS duplicates
FROM  nsr.stg_households;

-- Which codes appear more than once?
SELECT household_code, COUNT(*) AS occurrences
FROM  nsr.stg_households
GROUP BY household_code
HAVING COUNT(*) > 1
ORDER BY occurrences DESC
LIMIT 20;

-- Are there rows that are completely identical?
SELECT household_code, head_name, pmt_score, COUNT(*) AS n
FROM  nsr.stg_households
GROUP BY household_code, head_name, pmt_score
HAVING COUNT(*) > 1;

      

      

        
Count before you delete — understand what you have first

        Never write a DELETE for duplicates until you have run a SELECT that shows exactly what would be removed. The SELECT with HAVING COUNT(*) > 1 shows the scope; a second query showing the full rows for one of those codes shows the content. Only then do you decide: are these truly the same household (safe to merge), or do they have different pmt_scores (which one is correct?). That question determines whether you auto-deduplicate or escalate for human review.
      


      
Intermediate — Deduplication Strategies

      

        
sql – find duplicates

        
-- Find duplicate household codes
SELECT household_code, COUNT(*) AS cnt
FROM nsr.stg_households
GROUP BY household_code
HAVING COUNT(*) > 1
ORDER BY cnt DESC;

-- Keep only the most recently imported row
DELETE FROM nsr.stg_households
WHERE ctid NOT IN (
  SELECT MIN(ctid)
  FROM nsr.stg_households
  GROUP BY household_code
);

-- NIN deduplication (same NIN, different household)
SELECT nin, COUNT(DISTINCT household_id) AS household_count
FROM nsr.individuals
WHERE nin IS NOT NULL
GROUP BY nin
HAVING COUNT(DISTINCT household_id) > 1;

      

      

        
Two kinds of duplicates — different fixes

        Duplicate household codes in staging: multiple rows arrived in the CSV with the same code — keep the one with the latest loaded_at and discard the rest before promoting to production. The ctid technique (physical row address) is a PostgreSQL-specific shortcut for this; in a production pipeline, use a proper DISTINCT ON (household_code) ORDER BY loaded_at DESC CTE. Duplicate NINs across households: the same individual appears enrolled in two households — a real data quality problem meaning either a data-entry error (person was enumerated twice) or a fraud signal (NIN cloned). The query surfaces the conflict; resolution requires a manual review workflow, not an automatic DELETE. Emphasise to participants: never auto-delete on a suspicion of duplicate — mark for review, log in audit_log, escalate to a supervisor.

Governance is not paperwork — it is the set of rules, controls, and accountability mechanisms that determine who can do what with data, and how those actions are recorded. For a DBA working on the NSR, governance translates directly into technical decisions: which columns to encrypt, which tables need RLS, who can approve a bulk DELETE, and whether the audit log can be queried by an external regulator. The Nigeria Data Protection Act 2023 is the legal framework; the database controls are how you prove compliance.

Step-by-step guide

            Step 1 — Create the staging table (if not exists — idempotent DDL).

            Step 2 — Insert 5 test rows including: one with a NULL household_code, one duplicate, one with a bad pmt_score ('N/A'), and two valid unique rows.

            Step 3 — Run the validation report (UNION ALL of issue counts).

            Step 5 — Verify: SELECT from households for the two new codes; SELECT from data_lineage for the batch record.

            Step 6 — Run the entire pipeline again without changing anything. Verify row counts did not change (idempotency test).

Key teaching points

              TRUNCATE staging before re-run: clearing and re-loading staging is safer than trying to detect "new" rows — it keeps the pipeline stateless. DISTINCT ON vs GROUP BY: DISTINCT ON keeps one full row (the most recent by loaded_at) rather than just the key — needed because you want all columns, not just the duplicate key. ON CONFLICT DO UPDATE: upsert instead of DO NOTHING means a re-enumeration updates existing households with fresh data rather than silently skipping them. Transaction wrapping insert + lineage: if the lineage insert fails, the data insert rolls back too — you never have data without a lineage record.

What to look for in the plan — and what to do at scale

            (a) Join strategy: with 3 rows, PostgreSQL will almost certainly choose a Nested Loop or Hash Join. At 5 million rows, a Hash Join on lga_id is expected and correct — it builds a hash table of lgas (small), then probes it for each household row. (b) Index use: at 3 rows, a Seq Scan on both tables is normal — the planner correctly judges that reading 3 rows sequentially is faster than a B-tree traversal. At 5 million rows, you want an index: CREATE INDEX idx_hh_lga_active ON nsr.households (lga_id) WHERE is_active = TRUE; — a partial index covering only active households reduces index size and maintenance cost. (c) shared_blks_hit: if this equals shared_blks_read + shared_blks_hit and read is > 0 after the first run, data is being fetched from disk. Then the query with WHERE l.state_code = 'PL' only reads the Plateau partition — partition pruning eliminates 35 other states entirely. Combine with the partial index on lga_id and the query becomes O(PL_rows × log(lga_count)) instead of O(all_rows).

Definitions & Key Terms

      
Complete glossary of concepts, terms, and acronyms used across the programme

    

  


  

    
  


  


    
    

      
Core Database Concepts

      


        

          
ACID

          
The four guarantees every PostgreSQL transaction provides: Atomicity (all-or-nothing), Consistency (rules never violated), Isolation (transactions don't interfere), Durability (committed data survives crashes). WAL delivers Durability; MVCC delivers Isolation.

          
Ask participants: "Which ACID property would break if PostgreSQL crashed mid-INSERT without WAL?"

          Day 1
        


        

          
Relation

          
The formal term for a table in relational theory. A relation has a fixed set of named attributes (columns) and an unordered set of tuples (rows). PostgreSQL is a relational database management system (RDBMS).

          Day 1
        


        

          
Tuple

          
A single row in a table. In PostgreSQL, a tuple includes system columns (xmin, xmax, ctid) in addition to user columns. MVCC stores multiple versions of the same logical row as separate physical tuples.

          Day 1
        


        

          
Primary Key

          
A column or combination of columns whose values uniquely identify every row in a table. PostgreSQL automatically creates a unique B-tree index for every PRIMARY KEY constraint. A table may have at most one primary key.

          
In the NSR schema: household_id BIGSERIAL PRIMARY KEY — system-generated, never reused.

          Day 1
        


        

          
Foreign Key

          
A constraint that ensures a column's value matches a value in another table's primary key, enforcing referential integrity. INSERT or UPDATE that violates the constraint is rejected with an error. ON DELETE CASCADE propagates deletions automatically.

          
nsr.individuals(household_id) REFERENCES nsr.households(household_id) ON DELETE CASCADE

          Day 1
        


        

          
Referential Integrity

          
The guarantee that every foreign key value corresponds to an existing primary key value. Prevents orphan rows — e.g. an individual record that references a household that no longer exists.

          Day 1
        


        

          
Normalisation1NF / 2NF / 3NF

          
The process of structuring tables to reduce redundancy and avoid update anomalies. 1NF: atomic values, no repeating groups. 2NF: no partial dependency on composite key. 3NF: no transitive dependency — every non-key column depends only on the primary key.

          
NSR target is 3NF: lga_name lives in nsr.lgas, not repeated in every household row.

          Day 1
        


        

          
Schema

          
A named namespace inside a database that groups related tables, views, functions, and sequences. Used to separate concerns and apply permissions. PostgreSQL databases typically have a public schema by default; the NSR uses a dedicated nsr schema.

          
CREATE SCHEMA nsr; SET search_path = nsr, public;

          Day 1
        


        

          
Constraint

          
A rule enforced by the database engine on column values or row relationships. Types: PRIMARY KEY, FOREIGN KEY, UNIQUE, NOT NULL, CHECK. Constraints are verified on every INSERT and UPDATE, rejecting violating rows at the database level regardless of the application.

          Day 1
        


        

          
Sequence

          
A database object that generates monotonically increasing integers. SERIAL and BIGSERIAL column types automatically create a sequence and set the column default to nextval(). Sequences are crash-safe and gap-tolerant — rolled-back transactions consume sequence values permanently.

          Day 1
        


      

    


    
    

      
PostgreSQL Architecture

      


        

          
Postmaster

          
The parent process that starts when PostgreSQL initialises. It listens for incoming connections, spawns one backend process per client, and manages background worker processes (autovacuum, WAL writer, checkpointer, etc.).

          Day 1
        


        

          
Backend Process

          
A dedicated OS process forked by the postmaster for each client connection. Handles SQL parsing, query planning, and execution exclusively for that connection. All backends share the shared_buffers pool. Connection pooling (PgBouncer) reduces the cost of spawning backends.

          Day 1
        


        

          
shared_buffers

          
PostgreSQL's in-memory page cache shared by all backends. Pages read from disk are placed here; writes go here first. Recommended: 25% of RAM. Increasing this reduces disk I/O for frequently accessed data (hot pages).

          
SET shared_buffers = '4GB'; — requires restart.

          Day 1 · Day 6
        


        

          
work_mem

          
Memory allocated per sort operation or hash table within a single query. A complex query with multiple sorts can use work_mem several times simultaneously. Setting too high causes OOM; too low spills to disk. Tune per session for heavy analytic queries.

          
SET work_mem = '256MB'; — session-level, safe to raise temporarily.

          Day 1 · Day 6
        


        

          
maintenance_work_mem

          
Memory for maintenance operations: VACUUM, CREATE INDEX, ALTER TABLE ADD FOREIGN KEY. Should be significantly higher than work_mem since only one maintenance operation runs per process. Typical: 512 MB–2 GB.

          Day 1 · Day 6
        


        

          
effective_cache_size

          
A planner hint — not actual allocated memory — telling the query planner how much total memory (shared_buffers + OS page cache) is available for caching. Affects whether the planner favours index scans (assume data cached) or sequential scans. Set to ~75% of total RAM.

          Day 6
        


        

          
Tablespace

          
A named location on disk where PostgreSQL stores database objects. Allows large tables or indexes to be placed on faster storage (e.g. SSD) while keeping smaller tables on standard disk. Default tablespace: pg_default.

          
CREATE TABLESPACE fast LOCATION '/ssd/pg_data'; CREATE INDEX … TABLESPACE fast;

          Day 3
        


        

          
pg_hba.conf

          
PostgreSQL's host-based authentication configuration file. Controls which hosts can connect, which databases and users they can reach, and what authentication method is required (md5, scram-sha-256, trust, reject, cert). Processed top-to-bottom; first matching rule wins.

          
host nsr_db nsr_readonly 10.0.0.0/8 scram-sha-256

          Day 3
        


      

    


    
    

      
WAL, MVCC & Transactions

      


        

          
WALWrite-Ahead Logging

          
The mechanism that delivers Durability. Before any data page is modified on disk, the change is written sequentially to the WAL log. On crash, PostgreSQL replays WAL from the last checkpoint to recover. Sequential WAL writes are faster than random data-page writes — this is why WAL enables high write throughput.

          
Analogy: WAL is the transaction journal in a bank — the record comes before the vault changes.

          Day 1
        


        

          
Checkpoint

          
A point at which all dirty pages in shared_buffers are flushed to disk. After a checkpoint, WAL replay only needs to start from that point — not the beginning of the log. Frequent checkpoints reduce crash-recovery time but increase I/O. Controlled by checkpoint_completion_target and max_wal_size.

          Day 1 · Day 6
        


        

          
MVCCMulti-Version Concurrency Control

          
PostgreSQL's mechanism for allowing readers and writers to operate concurrently without blocking each other. Every UPDATE creates a new tuple version rather than overwriting; readers see the version visible at their transaction's snapshot time. This eliminates read-write lock contention.

          
Analogy: Git branches — old commits remain accessible even after new ones are added.

          Day 1
        


        

          
xmin / xmax

          
Hidden system columns on every PostgreSQL tuple. xmin: transaction ID (xid) that inserted this row version — visible from this xid onwards. xmax: xid that deleted or replaced it — 0 means the row is still live. The visibility rules engine uses these to determine what each transaction can see.

          Day 1
        


        

          
Snapshot Isolation

          
PostgreSQL's default isolation level (READ COMMITTED sees the latest committed row per statement; REPEATABLE READ sees a snapshot taken at transaction start). Readers never block writers and writers never block readers — each sees a consistent point-in-time view of the database.

          Day 1
        


        

          
Dead Tuple

          
A row version that is no longer visible to any active transaction — produced by UPDATE or DELETE under MVCC. Dead tuples occupy disk space and must be reclaimed by VACUUM. Accumulation of dead tuples causes table bloat and degrades sequential scan performance.

          Day 1 · Day 6
        


        

          
VACUUM

          
The PostgreSQL process that reclaims storage from dead tuples and advances the transaction ID horizon to prevent XID wraparound. VACUUM does not return space to the OS by default — VACUUM FULL rewrites the table and does. autovacuum runs VACUUM automatically in the background.

          
SELECT relname, n_dead_tup FROM pg_stat_user_tables ORDER BY n_dead_tup DESC;

          Day 1 · Day 6
        


        

          
autovacuum

          
A background daemon that automatically triggers VACUUM and ANALYZE on tables that exceed a configurable dead-tuple threshold. Essential for long-running production systems. Can be tuned per-table with storage parameters. Disabling it is almost never the right answer.

          
ALTER TABLE nsr.households SET (autovacuum_vacuum_scale_factor = 0.01);

          Day 6
        


        

          
XID Wraparound

          
PostgreSQL uses 32-bit transaction IDs. After ~2 billion transactions, IDs would wrap around, making old rows appear to be in the future. VACUUM prevents this by freezing old tuples. If wraparound is imminent, PostgreSQL enters a protective read-only shutdown mode.

          Day 6
        


        

          
Bloat

          
The accumulation of dead tuples and free space within a table or index that has not yet been reclaimed by VACUUM. Causes sequential scans to read more pages than necessary and indexes to become larger than optimal. Monitored via pgstattuple extension.

          Day 6
        


      

    


    
    

      
SQL & Query Language

      


        

          
DDLData Definition Language

          
SQL statements that define or alter database structure: CREATE, ALTER, DROP, TRUNCATE, RENAME. DDL statements in PostgreSQL are transactional — they can be rolled back within a BEGIN/COMMIT block, unlike many other databases.

          Day 2
        


        

          
DMLData Manipulation Language

          
SQL statements that read or modify data: SELECT, INSERT, UPDATE, DELETE, MERGE. DML operations are always transactional and subject to MVCC visibility rules.

          Day 2
        


        

          
DCLData Control Language

          
SQL statements that control access: GRANT and REVOKE. Used to assign or remove privileges on tables, schemas, sequences, and functions to roles. Works in conjunction with Row-Level Security for fine-grained access control.

          Day 2 · Day 3
        


        

          
JOIN

          
Combines rows from two tables based on a related column. INNER JOIN: only matching rows. LEFT JOIN: all left rows, NULLs where no match on right. RIGHT JOIN: inverse. FULL OUTER JOIN: all rows from both sides. CROSS JOIN: cartesian product.

          
Common trap: LEFT JOIN returning unexpected NULLs because the ON clause filters on the right-hand table — move such filters to WHERE only if you intend an inner join.

          Day 2
        


        

          
CTECommon Table Expression

          
A named subquery defined with the WITH clause, referenced one or more times in the main query. Improves readability for complex logic. In PostgreSQL 12+, non-recursive CTEs are inlined (optimised) by default; MATERIALIZED keyword forces separate execution.

          
WITH eligible AS (SELECT … FROM nsr.households WHERE pmt_score < 40) SELECT …

          Day 2
        


        

          
Window Function

          
An aggregate-like function that computes a value for each row using a set of related rows (the "window"), without collapsing the result to one row per group. Defined with OVER (PARTITION BY … ORDER BY …). Examples: ROW_NUMBER(), RANK(), LAG(), LEAD(), SUM() OVER.

          
ROW_NUMBER() OVER (PARTITION BY state_code ORDER BY pmt_score) — rank households within each state.

          Day 2
        


        

          
Correlated Subquery

          
A subquery that references a column from the outer query, causing it to be re-evaluated for each outer row. Often replaceable with a JOIN or window function for better performance. EXISTS with a correlated subquery is a common and efficient pattern.

          Day 2
        


        

          
UPSERTINSERT … ON CONFLICT

          
A PostgreSQL extension to INSERT that handles duplicate-key conflicts gracefully. ON CONFLICT DO NOTHING ignores duplicates; ON CONFLICT DO UPDATE (MERGE) updates the existing row. Essential for idempotent ETL pipelines loading the NSR.

          
INSERT INTO nsr.households … ON CONFLICT (household_code) DO UPDATE SET pmt_score = EXCLUDED.pmt_score;

          Day 2 · Day 7
        


        

          
EXPLAIN / EXPLAIN ANALYZE

          
EXPLAIN shows the query plan the planner chose without executing the query. EXPLAIN ANALYZE executes the query and shows actual timing and row counts alongside estimates. Key columns: cost, rows, width, actual time, loops. Use BUFFERS option to see cache hit/miss.

          
EXPLAIN (ANALYZE, BUFFERS, FORMAT TEXT) SELECT …

          Day 2 · Day 6
        


      

    


    
    

      
Indexes & Query Planning

      


        

          
B-tree Index

          
The default and most general PostgreSQL index type. Supports equality and range comparisons (<, <=, =, >=, >) and ORDER BY. Automatically maintained on INSERT/UPDATE/DELETE. Best for columns used in WHERE clauses and JOIN conditions with high or moderate selectivity.

          Day 6
        


        

          
GIN IndexGeneralised Inverted Index

          
Index type for multi-valued columns: arrays, JSONB, full-text tsvector. Stores a posting list of which rows contain each element. Faster for containment searches (@>, <@) and full-text @@ queries. Write-slower than B-tree; suitable for columns that are read far more than written.

          
CREATE INDEX ON nsr.individuals USING GIN (to_tsvector('english', full_name));

          Day 6
        


        

          
BRIN IndexBlock Range INdex

          
An extremely compact index that stores min/max values for ranges of physical blocks. Effective only on naturally ordered columns such as insertion timestamps or sequential IDs. Very low storage overhead; can replace a B-tree on append-only time-series tables with millions of rows.

          Day 6
        


        

          
Partial Index

          
An index built over a subset of rows, defined by a WHERE clause. Smaller and faster than a full index when queries consistently filter to the same subset. Ideal for NSR active-household queries where 95% of lookups target is_active = TRUE.

          
CREATE INDEX idx_active ON nsr.households (lga_id) WHERE is_active = TRUE;

          Day 6
        


        

          
Covering IndexINCLUDE clause

          
A B-tree index that includes additional non-key columns in its leaf pages, allowing index-only scans that never touch the heap. Added with INCLUDE in PostgreSQL 11+. Eliminates heap fetches for queries that only need the indexed and included columns.

          
CREATE INDEX ON nsr.households (lga_id) INCLUDE (head_name, pmt_score);

          Day 6
        


        

          
Sequential Scan

          
The planner reads every page of the table from start to finish. Preferred when the query returns a large fraction of rows (low selectivity) or the table is small enough to fit in shared_buffers. Not inherently bad — for full-table reads it is faster than random index lookups.

          Day 2 · Day 6
        


        

          
Index Scan vs Index-Only Scan

          
An index scan uses the index to find row locations, then fetches each row from the heap (table). An index-only scan satisfies the query entirely from the index leaf pages without heap access — possible only if all needed columns are in the index (covering index) and the visibility map shows the page is all-visible.

          Day 6
        


        

          
Selectivity

          
The fraction of rows an index condition is expected to match. High selectivity = few matching rows (e.g. WHERE nin = 'AB1234567CD') — index scan is efficient. Low selectivity = many rows match (e.g. WHERE gender = 'M') — sequential scan may be cheaper. The planner uses column statistics (pg_stats) to estimate selectivity.

          Day 6
        


        

          
pg_stats / ANALYZE

          
ANALYZE collects statistical histograms on column value distributions and stores them in pg_stats. The query planner uses these to estimate row counts and choose join/scan strategies. Stale statistics (after a bulk load without ANALYZE) cause poor query plans. autovacuum triggers ANALYZE automatically.

          
ANALYZE nsr.households; — run after bulk ETL loads.

          Day 6
        


        

          
Hash Join / Nested Loop / Merge Join

          
The three join algorithms PostgreSQL's planner chooses between. Hash join: hashes the smaller table, probes with the larger — good for large unsorted joins. Nested loop: for each outer row, scans inner — good when inner side is small or index-accessible. Merge join: both sides sorted — good for large sorted inputs or when results need to be ordered.

          Day 6
        


      

    


    
    

      
Administration, Security & Backup

      


        

          
Role

          
PostgreSQL's unified concept for both users and groups. A role with LOGIN privilege is a user; one without is a group. Roles can own database objects, be granted privileges, and be members of other roles. Best practice: create a group role with privileges, then grant users membership.

          
CREATE ROLE nsr_readonly; GRANT SELECT ON ALL TABLES IN SCHEMA nsr TO nsr_readonly; GRANT nsr_readonly TO caseworker_john;

          Day 3
        


        

          
RLSRow-Level Security

          
A PostgreSQL feature that restricts which rows a user can see or modify, enforced at the storage engine level — not the application. Policies are defined per table per command (SELECT, INSERT, UPDATE, DELETE). Once enabled, even superusers must use BYPASSRLS to bypass.

          
Policy: caseworkers in Lagos can only SELECT households WHERE state_code = current_setting('app.state_code').

          Day 3
        


        

          
pg_dump / pg_restore

          
pg_dump creates a logical backup of a database or specific objects in SQL script or custom/directory/tar format. pg_restore loads custom-format backups, allowing selective table or schema restore. Not a substitute for physical backup for large databases — use pg_basebackup for that.

          
pg_dump -Fc -d nsr_db -f nsr_backup.dump — custom format, compressed.

          Day 3
        


        

          
pg_basebackup

          
A physical backup tool that copies the entire PostgreSQL data directory while the server is running, using the streaming replication protocol. Produces a backup that can be restored to any point in time when combined with WAL archives. Required for PITR setups.

          Day 3
        


        

          
PITRPoint-In-Time Recovery

          
The ability to restore a database to any specific moment in the past by replaying WAL changes from a base backup up to the target time. Requires continuous WAL archiving (archive_mode = on). Essential for disaster recovery in production social-protection databases.

          
restore_command = 'cp /archive/%f %p' — copies archived WAL segments during recovery.

          Day 3
        


        

          
WAL Archiving

          
Copying completed WAL segment files to a durable external location (object storage, NFS, S3) as they are produced. Enables PITR and standby servers that may fall behind the primary. Configured via archive_mode = on and archive_command in postgresql.conf.

          Day 3
        


        

          
Streaming Replication

          
A high-availability mechanism where a standby server continuously receives and replays WAL records from the primary in near real-time. Standby can be used for read queries (hot standby). Synchronous replication guarantees zero data loss at the cost of write latency.

          Day 3
        


        

          
SSL / TLS

          
Transport-layer encryption for PostgreSQL connections. Configured with ssl = on in postgresql.conf and certificate/key files. pg_hba.conf hostssl lines require SSL from specific hosts. Mandatory for production systems handling PII such as the NSR. Client certificates can provide mutual authentication.

          Day 3
        


        

          
pgAudit

          
A PostgreSQL extension that provides detailed session and object-level audit logging. Logs which user, from which host, executed which statement on which object, and when. Required for compliance with GDPR, NDPA (Nigeria), and government data governance frameworks. Output goes to PostgreSQL's standard log.

          Day 3 · Day 7
        


      

    


    
    

      
Performance Tuning & Partitioning

      


        

          
Table Partitioning

          
Dividing a large table into smaller physical sub-tables (partitions) while presenting a single logical table to queries. PostgreSQL supports Range (e.g. by year), List (e.g. by state_code), and Hash (even distribution). The planner prunes irrelevant partitions — a query for Lagos never scans Kano's partition.

          
PARTITION BY LIST (state_code) — each Nigerian state gets its own partition.

          Day 6
        


        

          
Partition Pruning

          
The planner's ability to skip partitions that cannot contain rows matching the query's WHERE clause. Dramatically reduces I/O on partitioned tables. Requires the partition key to appear in the query predicate. Enable with enable_partition_pruning = on (default).

          Day 6
        


        

          
Connection PoolingPgBouncer

          
A proxy layer that maintains a pool of long-lived database connections and multiplexes many short-lived client connections on top. Eliminates the overhead of spawning a new backend process per request. PgBouncer is the most common tool; operates in session, transaction, or statement mode.

          
Without pooling: 1,000 app threads = 1,000 backend processes. With pooling: 1,000 threads sharing 20 backends.

          Day 6
        


        

          
random_page_cost

          
A planner cost constant representing the relative cost of fetching a non-sequentially accessed disk page. Default: 4.0 (spinning disk assumption). Lower it to 1.1–1.5 for SSDs to encourage the planner to use index scans more aggressively, since random reads are cheap on SSDs.

          
SET random_page_cost = 1.2; — appropriate for SSD-backed storage.

          Day 6
        


        

          
checkpoint_completion_target

          
Fraction of the checkpoint interval over which PostgreSQL spreads dirty page writes to disk. Default: 0.9 — spread writes over 90% of the interval to avoid I/O spikes. Lowering it causes faster but more I/O-intensive checkpoints.

          Day 6
        


        

          
pg_stat_statements

          
A PostgreSQL extension that records execution statistics for every query: total calls, total time, mean time, rows returned, cache hit ratio. The primary tool for identifying slow or frequently-called queries in production. Essential for performance baseline comparisons.

          
SELECT query, mean_exec_time, calls FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;

          Day 6
        


      

    


    
    

      
ETL/ELT & Data Governance

      


        

          
ETLExtract, Transform, Load

          
Data integration pattern where data is extracted from source systems, transformed in an intermediate layer (cleaning, deduplication, mapping), then loaded into the target database. Transformation happens before loading — the database receives clean data.

          
Field enumeration data (CSV) → Python pandas cleaning → pg COPY into nsr.households staging table → INSERT INTO final tables.

          Day 7
        


        

          
ELTExtract, Load, Transform

          
Data integration pattern where raw data is loaded into the database first, then transformed using SQL. Leverages the database engine's processing power. Preferred for modern cloud data warehouses and when the target system (PostgreSQL) is powerful enough to transform data efficiently.

          Day 7
        


        

          
COPY

          
PostgreSQL's high-speed bulk data import/export command. COPY FROM reads a file or STDIN into a table; COPY TO writes a table or query result to a file. Far faster than individual INSERTs — bypasses statement parsing overhead. \copy in psql runs client-side (no superuser needed).

          
COPY nsr.households FROM '/data/enumeration_round3.csv' CSV HEADER;

          Day 7
        


        

          
Staging Table

          
A temporary or permanent table that receives raw imported data before it is validated, deduplicated, and merged into production tables. Common pattern: COPY raw CSV into staging → SQL transformation → UPSERT into final tables. Allows re-running failed loads without corrupting production data.

          Day 7
        


        

          
FDWForeign Data Wrapper

          
A PostgreSQL extension mechanism for querying external data sources as if they were local tables. postgres_fdw connects to another PostgreSQL instance; file_fdw reads CSV files; oracle_fdw connects to Oracle. Pushes WHERE clauses to the remote server where possible.

          
CREATE SERVER cbms_server FOREIGN DATA WRAPPER postgres_fdw OPTIONS (host 'cbms-db', dbname 'cbms');

          Day 7
        


        

          
Data Lineage

          
The documented trail of where data originated, how it was transformed, and where it flows. Essential for social-protection systems to verify that a household's eligibility score (PMT) can be traced back to raw enumeration data. Enables audit, debugging, and compliance reporting.

          Day 7
        


        

          
GDPR / UK GDPR

          
General Data Protection Regulation — EU law (and its UK equivalent post-Brexit) governing personal data handling. Key obligations: lawful basis for processing, data minimisation, subject access rights, right to erasure, breach notification (72 hours). Nigerian equivalent: NDPA 2023.

          
Deleting a beneficiary on erasure request: logical deletion (is_active = FALSE) + audit log, not physical DELETE — preserves referential integrity.

          Day 7
        


        

          
NDPANigeria Data Protection Act 2023

          
Nigeria's primary data protection legislation, establishing the Nigeria Data Protection Commission (NDPC). Aligns broadly with GDPR principles: consent, purpose limitation, data minimisation, security. Directly applicable to the NSR, CBMS, and any government system processing citizens' personal data.

          Day 7
        


        

          
Data Retention Policy

          
A documented rule specifying how long each category of data is kept, in what form, and when it must be deleted or anonymised. Balances legal obligations (retain for audit) against privacy rights (erase when no longer needed). Implemented in PostgreSQL using scheduled jobs, partitioning, or logical soft-delete patterns.

          Day 7
        


        

          
Idempotency

          
A property of an operation where running it multiple times produces the same result as running it once. ETL pipelines should be idempotent: re-running a failed load must not create duplicate records or corrupt data. Achieved using UPSERT (INSERT … ON CONFLICT DO UPDATE) and staging tables.

          Day 7

NSRNational Social Registry

          
Nigeria's centralised household data platform managed by the National Social Safety Net Coordinating Office (NASSCO). Aggregates enumeration data from all states to determine household eligibility for social protection programmes. The core use case for this entire training programme.

PMT ScoreProxy Means Test

          
A statistical score computed from observable household characteristics (housing type, assets, demographics) as a proxy for income, which is hard to verify in low-documentation contexts. An 11-character alphanumeric identifier assigned to Nigerian citizens by NIMC (National Identity Management Commission). Format: 2 uppercase letters + 7 digits + 2 uppercase letters. Used in the NSR to deduplicate individuals across enumeration rounds and link beneficiary records.

A scheduled nationwide or state-level household survey exercise that captures or updates PMT data. The NSR stores the enumeration_round column on household records to track data vintage. Subsequent rounds update existing records (UPSERT) rather than inserting duplicates.
